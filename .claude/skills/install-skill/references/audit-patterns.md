# Audit patterns — спецификация

Этот файл — человекочитаемая спецификация чек-листа. **Авторитетный источник** — [`scripts/audit.sh`](../scripts/audit.sh): там фактические регулярки, и если кто-то их ломает — ломается именно поведение. Здесь — **что** проверяем и **почему**. Регулярки, если есть, иллюстративные.

При расхождении между этим документом и скриптом — **доверять скрипту**. Обнаружив расхождение, сначала обнови скрипт, затем подправь этот документ.

## Что сканируется

Текстовые файлы в корне скилла (рекурсивно), расширения:
`.md .py .sh .bash .zsh .fish .js .mjs .cjs .ts .tsx .jsx .json .yaml .yml .toml .ini .cfg`

Плюс отдельно — **бинарные файлы** через `file --mime-encoding` (не us-ascii / utf-8 / ascii / iso-8859 → бинарник → hard-stop). Скилл не должен таскать с собой исполняемые файлы: бинарники непроверяемы на глаз, и их обычно приносят для запуска на машине пользователя.

## Hard-stop (блокируют установку)

### 1. Чтение секретов
Зачем проверяем: скилл, который читает `.env`, `~/.ssh/`, `~/.aws/`, `credentials.json`, `.netrc` или зовёт `git config --get` для токенов — это **эксфильтрация**. Легитимный скилл не лезет в чужие ключи без прямого указания.

Триггеры: упоминание путей `.env` (не `.env.example`), `~/.ssh`, `/root/.ssh`, `~/.aws`, `~/.config/gcloud`, `credentials.{json,txt,yaml,yml}`, вызовы `keyring.*`, `git config --get`, чтение `.netrc`.

### 3. Деструктив
Зачем: команды, которые удаляют/затирают/перезаписывают данные пользователя без возможности отката.

Триггеры: `rm -rf`, `shred`, `dd ... of=/dev/`, `mkfs`, редирект в `/dev/sd*`, `git push --force` / `git push -f`, `git reset --hard`, `truncate -s 0`.

### 4. Выход за пределы каталога скилла
Зачем: скилл должен жить внутри `~/.claude/skills/<name>/` и взаимодействовать с проектом пользователя только по явным путям. Правка `~/.bashrc`, `/etc/*`, `~/.claude/settings.json` — это несогласованное изменение среды.

Триггеры: `../../`, абсолютные пути в `/etc/`, `/usr/(bin|local|lib)`, `/var/(log|lib|spool)`, `/root/`, упоминания `~/.bashrc`, `~/.zshrc`, `~/.profile`, `~/.gitconfig`, `~/.ssh/config`, `~/.claude/settings.json`.

### 5. Динамическое исполнение
Зачем: `eval`, `exec`, `subprocess(..., shell=True)`, `os.system`, `new Function(...)` — классические RCE-гаджеты, которые могут запустить что угодно в зависимости от входных данных. Дают атакующему ручку, даже если сам скилл выглядит безобидно.

Триггеры: вызовы `eval(...)`, `exec(...)`, `subprocess.*shell=True`, `os.system(...)`, `new Function(...)`, `Function(...)(...)`.

### 6. Повышение прав
Зачем: скилл не должен требовать root или трогать права у чужих файлов. `curl | sh` — хрестоматийная install-yolo-атака.

Триггеры: `sudo`, `chmod +x`, `curl ... | sh/bash/zsh`, `wget ... | sh/bash/zsh`, `chown`, `setuid`, `setgid`.

### 7. Обфускация
Зачем: `base64 -d | sh`, `eval(atob(...))`, минифицированный JS без исходников — почти всегда означают «что-то прячем». Легитимным скиллам скрывать нечего.

Триггеры: `base64 -d`, `base64 --decode`, `atob(...)`, `eval(atob(...))`. Дополнительная эвристика в скрипте: строки длиннее ~200 символов без пробелов (кандидаты в base64).

## Warn (требуют подтверждения пользователя)

### 2. Исходящая сеть
Зачем: скилл, который дёргает внешние URL, может отправлять данные наружу. Но это часто **легитимно** (fetch API, скачивание модели, проверка версии). Решение за пользователем.

Триггеры: `curl`, `wget`, `httpx`, `requests.(get|post|...)`, `fetch(...)`, `urllib.request`, `urlopen`, `aiohttp.*`.

### 8. Рантайм-зависимости
Зачем: `pip install` / `npm install`, запущенные **в момент использования скилла**, — это автоматическая инсталляция произвольного софта, возможно без подтверждения пользователя. Часто это легитимная инструкция в документации («Requires: pip install foo») — тогда warn безобиден. Но если это `subprocess.run(['pip', 'install', ...])` в рантайм-скрипте — уже серьёзно.

Триггеры: `pip install`, `pip3 install`, `npm install`, `pnpm install`, `yarn add`, `cargo install`, `go install`, `brew install`, `apt install`, `apt-get install`.

**Контекст важен.** Если совпадение в `SKILL.md` внутри fenced code block — чаще всего это документация. В `setup.sh` или в теле Python-скрипта — скорее, runtime-поведение. Скрипт не различает контекст автоматически, это делает человек при просмотре отчёта.

## False positives и исключения

Некоторые паттерны встречаются в **документации** со смыслом «не делай так»:

- fenced code block после подписи `BAD:`, `DON'T:`, `Anti-pattern:`, `Never:`
- комментарий `# example of what NOT to do`, `// unsafe example`

Скрипт найдёт такие совпадения и покажет — **это правильное поведение, не баг**. Решение принимает человек. Если совпадение явно в doc-контексте — можно подтвердить warn и продолжить. Если hard-stop — не обходится, даже если «это же в документации»: безопасность важнее удобства.

**Special case: self-install.** Если устанавливаемый скилл — сам `install-skill`, скрипт неизбежно сработает, потому что regex-ы паттернов живут в нём как текст. SKILL.md предписывает пропустить аудит для self-install, но всё равно спросить пользователя и убедиться, что источник — github-URL, которому доверяем.

## Как обновлять список

При добавлении/изменении паттерна:
1. Правь [`scripts/audit.sh`](../scripts/audit.sh) — это код, который реально исполняется.
2. Обнови соответствующий раздел здесь коротко: «что ищем и зачем».
3. Прогони аудит на известном эталонном скилле (например, `anthropics/skills/pdf`) — убедись, что не появилось новых false-positive.
