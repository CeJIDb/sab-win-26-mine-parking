# install-skill

Берёт GitHub-URL, клонирует репо во временный каталог, проверяет содержимое по чек-листу опасных паттернов, спрашивает у пользователя место установки (глобально или в текущий проект) и только после этого копирует скилл на постоянное место. Если в коде есть «красные» паттерны — отказ; «жёлтые» — показываем и просим подтверждение.

Цель скилла — убрать рутину «скачал-посмотрел-скопировал» и не пустить в `~/.claude/skills/` что-то очевидно опасное, не читая внимательно каждый файл руками.

## Когда срабатывать

Триггер — пользователь хочет установить скилл из GitHub. Примеры:

- «установи скилл https://github.com/anthropics/skills/tree/main/pdf»
- «поставь скилл по этой ссылке: <url>»
- «добавь мне этот скилл <url>»
- «install skill <url>»
- «затащи к себе <url>»
- «вот скилл — поставь» + GitHub-ссылка в сообщении

Если пользователь произнёс слово «скилл» вместе с GitHub-ссылкой и глаголом-намерением (установить / поставить / добавить / затащить / инсталлировать) — запускайся.

Если пользователь говорит «установи» + GitHub-ссылку без слова «скилл», но в ссылке есть `SKILL.md` или путь `/skills/` — тоже запускайся.

## Ограничения

- Работаем **только** с `https://github.com/*`. Другие хосты (GitLab, Bitbucket, прямой ZIP) — отказ, объясни пользователю границу и предложи дать github-эквивалент.
- Не запускаем никаких скриптов из скачиваемого скилла во время установки. Только чтение + копирование.
- Не правим PATH, не создаём симлинки, не трогаем `~/.bashrc`, `~/.zshrc`, `~/.claude/settings.json`.
- Не ставим бинарные файлы. Если в скилле есть бинарник — это hard-stop.

## Алгоритм

### Шаг 1. Получить URL

Если пользователь не дал URL (просто сказал «установи скилл pdf») — спроси прямо:

> Дай GitHub-URL скилла (`https://github.com/...`). Без ссылки не ставлю.

Если URL не на `github.com` — остановись, объясни: «Скилл работает только с github.com, scope осознанно ограничен. Дай github-эквивалент или поставь руками».

### Шаг 2. Распарсить URL

Допустимые формы:

- `https://github.com/<owner>/<repo>` — скиллом считается корень репо
- `https://github.com/<owner>/<repo>/tree/<ref>/<subpath>` — скилл в подпапке
- `https://github.com/<owner>/<repo>/blob/<ref>/<subpath>/SKILL.md` — родитель пути и есть скилл

Извлеки: `owner`, `repo`, `ref` (если нет — `main`), `subpath` (может быть пустым).

### Шаг 3. Клонировать во временный каталог

```bash
TMP="$(mktemp -d -t install-skill.XXXXXX)"
git clone --depth 1 --branch "<ref>" "https://github.com/<owner>/<repo>.git" "$TMP" 2>&1
```

`mktemp -d` гарантирует уникальность и пишет туда, куда у процесса есть права (обычно `/tmp`, но может быть `$TMPDIR`).

Если указанная ветка не существует — попробуй `main`, потом `master`. Если репо вообще недоступен — покажи ошибку, прерви установку.

### Шаг 4. Найти корень скилла

- Если был `subpath` → `SKILL_DIR="$TMP/<subpath>"`
- Иначе → `SKILL_DIR="$TMP"`

Проверь, что `$SKILL_DIR/SKILL.md` существует.

**Если `SKILL.md` по указанному пути нет** — не молчи и не прерывайся сразу. Сделай поиск по всему репо и покажи пользователю варианты:

```bash
find "$TMP" -name 'SKILL.md' -not -path '*/.git/*' | sed "s|^$TMP/||; s|/SKILL.md\$||"
```

Сформируй сообщение:

> В `<subpath>` нет `SKILL.md`. Но в репо `<owner>/<repo>` есть скиллы в других папках:
>
> - `skills/pdf`
> - `skills/docx`
> - `skills/xlsx`
>   ...
>
> Который ставим? (ответь номером или путём, или «отмена»)

Если пользователь выбирает — переопредели `subpath` и повтори шаг 4. Если отменяет — очисти `$TMP`, стоп.

Если в репо вообще нет ни одного `SKILL.md` — это не скилл-репо. Скажи:

> В `<owner>/<repo>` я не нашёл ни одного `SKILL.md`. Это репо точно про скиллы?

### Шаг 5. Извлечь имя скилла из SKILL.md

Из YAML-frontmatter достань поле `name:`. Это имя, под которым будем ставить.

Если `name:` нет в frontmatter — возьми имя последней непустой части URL (имя подпапки или имя репо). Предупреди пользователя, что имя выведено из URL, а не из файла.

### Шаг 6. Спросить место установки

Задай вопрос буквально так:

> Куда ставить `<name>`?
>
> - **Глобально** — `~/.claude/skills/<name>/` — будет доступен во всех проектах.
> - **В текущий проект** — `<cwd>/.claude/skills/<name>/` — только в этом проекте.

Дождись ответа. Варианты ответа «глобально», «global», «g», «1» → глобально. «проект», «project», «p», «2», «локально» → в проект.

Зафиксируй `TARGET_DIR`.

### Шаг 7. Проверить конфликт имён

**7a. Конфликт в целевом scope.** Если `TARGET_DIR` уже существует — спроси:

> Скилл `<name>` уже установлен в `<TARGET_DIR>`. Что делаем?
>
> - **обновить** — перезаписать свежей версией (старая сохранится в `<skills-root>/.backups/<name>-<YYYYMMDD-HHMMSS>/`)
> - **пропустить** — оставить текущую версию, ничего не делать
> - **прервать** — остановить установку

На «пропустить» / «прервать» — очисти `$TMP`, остановись. На «обновить» — продолжай, бэкап сделаешь на шаге 10.

**7b. Двойной scope-конфликт (важно для избежания дрифта).** Проверь и **другой** scope:

- если ставим глобально (`~/.claude/skills/<name>/`) → проверь, есть ли `<cwd>/.claude/skills/<name>/`
- если ставим в проект (`<cwd>/.claude/skills/<name>/`) → проверь, есть ли `~/.claude/skills/<name>/`

Если найдено — **не молчи**, предупреди:

> `<name>` уже стоит в другом scope: `<other-path>`. Если сейчас поставлю ещё один экземпляр, Claude увидит оба в `available_skills` с одинаковым именем — это работает, но **версии могут разойтись**. Продолжаем? (да/нет)

На «нет» — стоп, `$TMP` почисти. На «да» — продолжай, но в финальном отчёте (шаг 11) напомни: «дубликат с `<other-path>`, синхронизируй при правках».

### Шаг 8. Security-аудит

Запусти авторитетный чек через скрипт:

```bash
bash "$(dirname "$0")/scripts/audit.sh" "$SKILL_DIR"
```

Если запускаешься через Skill tool и не знаешь `$0`, используй полный путь к скрипту — `~/.claude/skills/install-skill/scripts/audit.sh` (глобальный) или `<cwd>/.claude/skills/install-skill/scripts/audit.sh` (проектный). Выбери тот, из которого SKILL.md тебя и запустил.

Скрипт возвращает структурированный отчёт со счётчиками `hard_stop`, `warnings`, `binaries` и код выхода:

- **0** — чисто или только warnings
- **1** — найдены hard-stop (включая бинарники)
- **2** — ошибка запуска (плохой путь)

Полный список паттернов и логика — в [references/audit-patterns.md](references/audit-patterns.md) (это спецификация для людей, правда authoritative-источник — сам скрипт).

**Важно про self-install.** Если устанавливаемый скилл — это сам `install-skill` (по любой причине: обновление, миграция), аудит даст много false-positive: в `scripts/audit.sh` и `references/audit-patterns.md` regex-ЛИТЕРАЛЫ hard-stop паттернов встречаются как текст. В этом случае **пропусти аудит** и установи напрямую — но сначала спроси пользователя, точно ли он хочет обновить сам `install-skill`, и откуда (github-URL обязан быть).

### Шаг 9. Решить по итогам аудита

- **Hard-stop ≥ 1** → установка прервана. Распечатай отчёт, скажи:

  > Установка прервана — в скилле есть паттерны из hard-stop списка. Если уверен в скилле и хочешь всё равно поставить — сделай `cp -r` руками, я больше ничего не трогаю.

  Очисти `$TMP`, стоп.

- **Только warnings** → покажи каждый warning отдельно и спроси «Продолжить установку? да/нет». На «нет» — прерви, очисти `$TMP`.

- **Чисто** → молча идём дальше, отчёт можно не показывать (или показать одной строкой «Проверка пройдена, подозрительных паттернов не найдено»).

### Шаг 10. Установка

Если это обновление существующего — сделай бэкап:

```bash
BACKUP_ROOT="$(dirname "$TARGET_DIR")/.backups"
TS=$(date +%Y%m%d-%H%M%S)
mkdir -p "$BACKUP_ROOT"
mv "$TARGET_DIR" "$BACKUP_ROOT/<name>-$TS"
```

Скопируй скилл на место:

```bash
mkdir -p "$(dirname "$TARGET_DIR")"
cp -r "$SKILL_DIR" "$TARGET_DIR"
```

Проверь, что `$TARGET_DIR/SKILL.md` на месте и читаемо.

### Шаг 11. Показать, как использовать

Прочитай свежеустановленный `$TARGET_DIR/SKILL.md`. Из frontmatter возьми `name` и `description`. Выведи пользователю:

```
Скилл `<name>` установлен в <TARGET_DIR>.

Что делает: <первое предложение description>

Попробуй:
  - «<пример триггерной фразы 1>»
  - «<пример триггерной фразы 2>»
  - «<пример триггерной фразы 3>»
```

Триггерные фразы ищи внутри `description` (часто они там перечислены через запятую или в кавычках) или сформулируй 2–3 правдоподобных на основе сути скилла. Если в description явно упомянуты русские фразы-триггеры — бери оттуда.

### Шаг 12. Уборка

```bash
rm -rf "$TMP" 2>/dev/null || true
```

Временный каталог удаляй всегда в самом конце, даже если установка провалилась.

**Fallback.** В некоторых средах (Claude Code sandbox, ограниченные контейнеры) `rm -rf` может быть запрещён из политики безопасности. Если удаление не прошло — **не паникуй и не повторяй**. Покажи пользователю одно сообщение:

> Временный каталог `<TMP>` оставлен (sandbox запретил `rm -rf`). Он в `$TMPDIR` / `/tmp` — система почистит его сама при следующей загрузке. Если хочешь прямо сейчас — удали руками: `rm -rf <TMP>`.

И продолжай дальше. Эта «недоуборка» не блокирует успешную установку — скилл уже на месте.

## Что делать, если пользователь настаивает в обход аудита

Пользователь может сказать «я знаю, что там `rm -rf`, мне норм, ставь». Это легитимный сценарий, но **этот скилл его не обслуживает**. Ответ:

> Hard-stop обходить нельзя — это защита от случайной ошибки. Если ты действительно уверен, скопируй руками: `cp -r <TMP>/<subpath> <TARGET_DIR>`. Я намеренно не делаю обход по запросу, чтобы эта черта оставалась жёсткой.

Warnings, наоборот, штатно обходятся через «да» — это нормальный режим.

## Приоритеты на случай конфликта инструкций

1. Безопасность аудита (hard-stop не обходится)
2. Явное подтверждение пользователя (на warn и на обновление)
3. Удобство (дефолты, автоопределение)

Если сомневаешься — спроси пользователя короткий вопрос, не додумывай.
