# plan-validator

Скилл для проверки, что файл плана в `plans/` соответствует правилам из `CLAUDE.md` (раздел «ВАЖНО: план для каждой новой функции») и `plans/README.md`.

## Когда использовать

- Пользователь только что создал новый файл `plans/YYYY-MM-DD-*.md` — проверь перед коммитом.
- Пользователь правил существующий план — убедись, что он всё ещё валиден.
- Пользователь явно просит: «проверь план», «валидируй план», «посмотри, соответствует ли план правилам».
- Перед коммитом в ветке с изменениями в `plans/`.

## Что проверяется

Пять правил, вытащенных из `CLAUDE.md` и `plans/README.md`:

1. **Имя файла** — `YYYY-MM-DD-slug.md`, где `YYYY-MM-DD` — валидная календарная дата, `slug` — kebab-case на латинице (`a-z`, `0-9`, одиночные дефисы, без двойных и хвостовых дефисов). Файл `README.md` в `plans/` не валидируется.
2. **H1-заголовок** — первая не-пустая строка содержательного контента начинается с `# ...`.
3. **Фазы со статусом** — в плане есть хотя бы одна строка, содержащая маркер `[ ]` или `[x]`. Это может быть заголовок фазы (`### Фаза 1. ... [x]`) или пункт чек-листа (`- [ ] Сделать X`).
4. **Итоговый блок** — в плане есть секция `## Итог` (или `## Итоги`), после заголовка идёт пустая строка, затем непустой контент.
5. **Предупреждение (warning, не ошибка)** — файл > 500 строк. Подозрение, что в одном плане смешалось несколько задач (правило «один план = одна задача»).

## Как запускать

### Режим 1 — автоматический (скрипт)

Каноничный валидатор живёт в `scripts/validate-plans.mjs` (версионируется вместе с репозиторием, используется husky pre-commit hook). Из корня проекта:

```bash
# Проверить все планы
npm run check:plans

# Проверить только то, что в git staged (то же делает pre-commit)
npm run check:plans:staged

# Проверить конкретный файл напрямую
node ./scripts/validate-plans.mjs plans/2026-04-24-tooling-adoption.md
```

Скрипт печатает итог по каждому файлу. Код выхода: `0` если всё ок, `1` если найдены ошибки, `2` если ошибка запуска.

### Режим 2 — Claude-агентом (если Node недоступен или скрипт не нашёлся)

Прочитай указанный файл плана целиком и пройди по чек-листу из раздела «Что проверяется» выше. Для каждого пункта отметь статус (✅/❌/⚠️) и короткое объяснение. В конце — сводка «N ошибок, M предупреждений».

## Формат отчёта (обе режима)

```text
✅ plans/2026-04-23-foo.md: OK

❌ plans/2026-04-23-bar.md:
   • Не найдено ни одной строки с [ ] или [x] — фазы без статуса
   • Нет секции `## Итог` в конце плана

Всего: 1 файл с ошибками.
```

## Что делать при каждой ошибке

- **Имя файла не по шаблону** → переименуй через `git mv old.md YYYY-MM-DD-slug.md`, чтобы сохранить историю.
- **Невалидная дата в имени** → проверь, что дата существует (например, не `2026-02-30`).
- **Нет H1** → добавь в начало файла строку `# План: <о чём план>`.
- **Нет фаз со статусом** → добавь раздел `## Фазы` со списком `- [ ] Фаза 1. ...`.
- **Нет `## Итог` или он пустой** → добавь в конец файла:

  ```markdown
  ## Итог

  <статус реализации: целиком / частично / не начато> + <что осталось>.
  ```

- **После `## Итог` нет пустой строки** → между заголовком и текстом должна быть пустая строка (`## Итог\n\nТекст`, а не `## Итог\nТекст`).
- **Предупреждение о >500 строк** → не ошибка, но подумай, не следует ли разделить план на 2–3 файла по задачам.

## Связанные файлы проекта

- `CLAUDE.md` — раздел «Технические планы» и «Как работать с этим репозиторием».
- `plans/README.md` — те же правила в развёрнутом виде.
- `scripts/check-branch-name.mjs` — стилистически аналогичный валидатор для имён веток (ориентир по стилю Node-скриптов в проекте).
