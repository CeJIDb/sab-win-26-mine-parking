## 1. RED: базовое поведение без инструкций

Ниже — типичные сбои, которые даёт агент **без** чётких инструкций субагента (что призваны предотвратить ваши промпты).

| Субагент | Ожидаемые сбои без инструкций (RED) |
|----------|-------------------------------------|
| **frontend-ux-ui** | Правка `wireframe.css` напрямую; игнорирование layout’ов (lk/admin/guard/site); дублирование разметки/стилей; отсутствие сборки; широкий поиск по всему репозиторию; слабая доступность (нет aria, семантики). |
| **git-workflow** | Коммит без явного запроса; сообщения на английском; `git add .` и коммит всего подряд; предложение/выполнение `push --force`; коммит секретов или файлов из `.gitignore`. |
| **meta-instructor** | Создание файлов в `docs/agents/`; инициативное изменение инструкций других субагентов; размытые формулировки («делай хорошо»); отсутствие раздела «Как использовать»; длинные промпты без привязки к путям проекта. |
| **systems-analyst** | Удаление требований без запроса; добавление требований без идентификаторов; нарушение структуры `docs/specs/*.md`; сканирование всего `docs/`; смешение роли с разработкой/архитектурой. |
| **doc-to-markdown** | Додумывание содержимого; чтение лишних файлов репозитория; избыточная структура или вымышленные метаданные/авторы/даты. |
| **wireframe-from-artifacts** | Игнорирование артефактов; правка `wireframe.css` вручную вместо сборки; широкий поиск по репозиторию; отсутствие связи разметки с артефактами (комментарии); путаница с frontend-ux-ui (когда какой вызывать). |

## 2. GREEN: покрытие инструкциями

Проверка: **устраивают ли текущие инструкции перечисленные сбои?**

### frontend-ux-ui ✅ в целом покрыт

- Явно: только SCSS, не трогать `wireframe.css`; список layout’ов и путей; UX (доступность, семантика, консистентность); переиспользование; сборка (`build-templates.mjs`, `npm run build:css`); ограничение контекста (Read/Grep/Glob по нужным путям).
- **Пробел:** нет явного «MUST NOT редактировать wireframe.css» в одном месте-правиле (есть разбросано).

### git-workflow ✅ покрыт

- Режим по умолчанию — только план, без выполнения; русский язык сообщений; запрет force и глобального config; запрет коммита секретов и файлов из `.gitignore`.
- **Пробел:** при давлении («срочно закоммить») агент может рационализировать и выполнить без явного запроса — нет явного анти-рационализации.

### meta-instructor ✅ покрыт

- Запрет создавать `docs/agents`; изменения только по прямому запросу; запрет инициативных правок и «косвенного» изменения через себя; структура файла с «Как использовать» и экономией токенов.
- **Пробел:** «прямой запрос» не формализован — при формулировках вроде «улучши субагентов» агент может решить, что можно менять инструкции.

### systems-analyst ✅ покрыт

- Не удалять требования без запроса; помечать «устарело»/«заменено»; идентификаторы (FR-xxx, NFR-xxx); сохранение структуры; работа только с указанными файлами.
- **Пробел:** нет явной отсылки к правилам в `.cursor/rules/docs.mdc` (редактирование спецификаций).

### doc-to-markdown ✅ покрыт

- Не придумывать содержимое; только переданные документы; стандартный Markdown; опциональный YAML; пометки о неточностях.
- **Пробел:** нет явного «MUST NOT добавлять факты/даты/авторов, которых нет в источнике».

### wireframe-from-artifacts ⚠️ частично

- Опора на артефакты; Nunjucks/SCSS; комментарии для связи с артефактами; ограничение контекста.
- **Пробел:** в п.3 указано «Генерировать (или обновлять) базовый CSS (например, ui/wireframe.css) на этапе сборки» — можно понять как «разрешено править wireframe.css». У frontend-ux-ui явный запрет правки. Нужна согласованность.

## 3. REFACTOR: рекомендации

### 3.1 Закрыть пробелы (как «закрыть лазейки»)

| Файл | Добавить / изменить |
|------|----------------------|
| **frontend-ux-ui.md** | В разделе «Поведение» или «Технологический стек» одну строку: **«Запрещено редактировать `ui/wireframe.css` напрямую; итоговый CSS только через сборку SCSS.»** |
| **git-workflow.md** | В «Поведение и режимы» добавить: **«Даже при срочности или просьбе „быстро закоммить“ не выполнять `git add`/`commit`/`push` без явной формулировки пользователя о выполнении команд (не только план).»** |
| **meta-instructor.md** | В «Ограничения» уточнить: **«Прямой запрос = явное упоминание обновления/изменения инструкции или файла в `.cursor/agents/` (например: „обнови инструкцию X“, „измени meta-instructor“). Общие просьбы вроде „улучши субагентов“ без указания конкретного файла не дают права менять `.cursor/agents/*.md».»** |
| **systems-analyst.md** | В раздел про `docs/specs` добавить: **«При правке спецификаций учитывай правила из `.cursor/rules/docs.mdc` (структура, идентификаторы, не удалять без пометки).»** |
| **doc-to-markdown.md** | В «Область ответственности» или «Формат ответа»: **«Не добавлять в документ факты, даты, авторов или ссылки, отсутствующие в переданном источнике.»** |
| **wireframe-from-artifacts.md** | В п.3 заменить формулировку про `wireframe.css` на: **«Итоговый CSS (в т.ч. `ui/wireframe.css`) получать через скрипт сборки (например `npm run build:css`); не редактировать `wireframe.css` вручную.»** — для согласованности с frontend-ux-ui. |

### 3.2 Ясность и структура

- **frontend-ux-ui / wireframe-from-artifacts:** различие «когда какой субагент» уже есть в frontend-ux-ui; в wireframe-from-artifacts можно в «Как использовать» одну строку: «Не использовать для точечной доработки готовых экранов по запросу пользователя — для этого frontend-ux-ui».
- **meta-instructor:** в начале «Формат ответа» явно перечислить обязательные блоки (переформулировка задачи → субагенты → мета-инфо → полное содержимое каждого `.cursor/agents/<name>.md`), чтобы агент не пропускал шаги.

### 3.3 Экономия токенов

- Общие принципы (ограничение контекста, Read/Grep перед SemanticSearch) уже есть у всех.
- **meta-instructor:** можно сократить повторяющийся блок про инструменты (Read, Grep, SemanticSearch, Glob, Shell) до одного общего списка с отсылкой «встраивать в создаваемые инструкции по необходимости».
- **systems-analyst:** блок «Контекст проекта» оставить как есть — он короткий и задаёт границы; дополнительно не раздувать.
