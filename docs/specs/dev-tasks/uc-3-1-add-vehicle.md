# Постановка задачи: Профиль клиента - Форма добавления ТС - Реализовать добавление ТС

Код UC: `UC-3.1`

Use Case: [Реестр Use Case](../../artifacts/use-case/use-case-registry.md), строка `UC-3.1 | Добавить ТС`

Схема БД: [chartdb-postgresql-erd-normalized-public.sql](../../../sql/database/chartdb-postgresql-erd-normalized-public.sql)

## Требование

Добавить ТС.

## Декомпозиция по компонентам архитектуры

### База данных

Создать таблицу:

- `vehicles`

и использовать существующие сущности:

- `vehicle_types`
- `clients`

с атрибутами и связями согласно прилагаемой физической схеме БД.

### Фронтенд

1. Создать макеты для формы добавления ТС в ЛК клиента на экране управления транспортными средствами на основе прототипов и согласовать с аналитиком. В макете предусмотреть поля:
   - ГРЗ;
   - марка;
   - модель;
   - цвет;
   - тип ТС.
2. Сверстать форму добавления ТС по согласованным макетам.
3. При вводе ГРЗ валидировать значение по реальным форматам государственных регистрационных знаков. Целевая аудитория — клиенты с ТС, зарегистрированными в РФ, странах СНГ и Европы. Источники: ГОСТ Р 50577-2018 (РФ); СТБ 977-94 и СТБ 1758-2007 (Беларусь); национальные стандарты по странам.

   **Базовые ограничения на поле ГРЗ:**

   | Параметр                    | Значение                                                                                                                              |
   | --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
   | Тип данных                  | строка (`VARCHAR(32)` в БД, см. `vehicles.license_plate`)                                                                             |
   | Длина после нормализации    | `5 <=` длина `<= 12`                                                                                                                  |
   | Допустимые символы          | заглавные латинские буквы `A–Z`, заглавные кириллические из набора ГОСТ `А В Е К М Н О Р С Т У Х`, цифры `0–9`, дефис `-`, пробел ` ` |
   | Нормализация при сохранении | привести буквы к верхнему регистру; обрезать ведущие и конечные пробелы; внутренние пробелы и дефисы сохранять как введено            |
   | Обязательность              | +                                                                                                                                     |
   | Уникальность                | глобальная по `vehicles.license_plate` (`UNIQUE` в БД); проверка на бэкенде до сохранения                                             |

   **Допустимые буквы для номеров РФ (ГОСТ Р 50577-2018):**

   Только 12 букв, имеющих визуальный аналог в латинице: кириллические `А В Е К М Н О Р С Т У Х`, или эквивалентные латинские `A B E K M H O P C T Y X`. Остальные кириллические буквы (Б, Г, Д, Ж, З, И, Й, Л, П, Ф, Ц, Ч, Ш, Щ, Ы, Ь, Э, Ю, Я) — недопустимы. Регулярное выражение для буквы РФ: `[АВЕКМНОРСТУХABEKMHOPCTYX]` (далее обозначается `L_RU`).

   **Поддерживаемые форматы (валидация регулярными выражениями):**

   В regex-шаблонах: `L_RU` — буква из набора ГОСТ выше; `\d` — цифра `0-9`; `[A-Z]` — латинская заглавная. Все шаблоны применяются к нормализованному значению (после `trim` и приведения к верхнему регистру).

   | Страна, тип ТС                   | Regex                                       | Пример                  |
   | -------------------------------- | ------------------------------------------- | ----------------------- |
   | РФ, легковой (тип 1)             | `^L_RU\d{3}L_RU{2}\d{2,3}$`                 | `А123ВС77`, `А123ВС777` |
   | РФ, прицеп (тип 2)               | `^L_RU{2}\d{4}\d{2,3}$`                     | `АА1234 77`             |
   | РФ, мотоцикл, трактор (тип 3, 4) | `^\d{4}L_RU{2}\d{2,3}$`                     | `1234АВ77`              |
   | РФ, военные (тип 7)              | `^\d{4}L_RU{2}\d{2}$`                       | `1234АА50`              |
   | РФ, полиция (тип 8)              | `^L_RU\d{4}\d{2,3}$`                        | `А1234 77`              |
   | Беларусь, до 2018                | `^\d{4}[ABCEHIKMOPTX]{2}-[1-7]$`            | `1234АВ-7`              |
   | Беларусь, с 2018                 | `^[ABCEHIKMOPTX]{2}\d{4}-[1-7]$`            | `АВ1234-7`              |
   | Казахстан                        | `^\d{3}[A-Z]{3}\d{2}$`                      | `123ABC02`              |
   | Украина                          | `^[ABCEHIKMOPTX]{2}\d{4}[ABCEHIKMOPTX]{2}$` | `АА1234ВВ`              |
   | Армения                          | `^\d{2} ?[A-Z]{2} ?\d{3}$`                  | `99 AB 123`             |
   | Кыргызстан                       | `^\d{2}[A-Z]{2}\d{3}[A-Z]{2}$`              | `01KG123AB`             |
   | Узбекистан                       | `^\d{2}[A-Z]\d{3}[A-Z]{2}$`                 | `01A123AB`              |
   | Молдова                          | `^[A-Z]{3}\d{3}$`                           | `ABC123`                |
   | Грузия                           | `^[A-Z]{2}-?\d{3}-?[A-Z]{2}$`               | `AB-123-CD`             |
   | Германия                         | `^[A-ZÄÖÜ]{1,3}-[A-Z]{1,2} ?\d{1,4}[HE]?$`  | `B-AB 1234`, `M-X 99`   |
   | Великобритания (с 2001)          | `^[A-Z]{2}\d{2} ?[A-Z]{3}$`                 | `BD51 SMR`              |
   | Франция (SIV, с 2009)            | `^[A-Z]{2}-\d{3}-[A-Z]{2}$`                 | `AB-123-CD`             |
   | Италия (с 1994)                  | `^[A-Z]{2} ?\d{3} ?[A-Z]{2}$`               | `AB123CD`, `AB 123 CD`  |
   | Польша                           | `^[A-Z]{2,3} ?[A-Z0-9]{4,5}$`               | `WX 12345`, `LU 1234A`  |
   | Испания (с 2000)                 | `^\d{4} ?[A-Z]{3}$`                         | `1234 BBC`              |
   | Латвия                           | `^[A-Z]{2}-\d{1,4}$`                        | `AB-1234`               |

   Допустимые латинские буквы для Беларуси, Украины и ряда стран СНГ ограничены набором, имеющим визуальные аналоги в кириллице: `A B C E H I K M O P T X`.

   **Логика валидации:**
   - **Фронтенд (мягкая валидация в реальном времени):** проверять только базовые ограничения из таблицы выше — длина 5–12 после нормализации, разрешенные символы, обязательность. Это позволяет вводить ГРЗ из любой страны без ошибочных блокировок неизвестных форматов.
   - **Бэкенд (строгая валидация при сохранении):** значение должно соответствовать как минимум одному regex из таблицы форматов выше; иначе вернуть ошибку с указанием, что формат не распознан. Дополнительно проверяется уникальность.

   **Источник базовых ограничений:** словарь данных, сущность «ТС», атрибут «грз» — [data-dictionary-buildin-import.md](../../architecture/database/erd/data-dictionary-buildin-import.md). Длина и набор символов в этом dev-task расширены относительно словаря для поддержки иностранных форматов и согласованы с физической схемой БД (`license_plate VARCHAR(32)`); словарь данных уточняет общий допустимый набор символов, а в этом dev-task зафиксированы реальные форматы государственных стандартов.

4. Поле выбора типа ТС отобразить в виде выпадающего списка на основе справочника `vehicle_types`.
5. При отправке формы передавать на бэкенд данные нового ТС, привязанные к текущему клиенту.
6. При успешном ответе бэкенда показывать пользователю сообщение об успешном добавлении ТС.
7. После показа сообщения об успехе выполнять перезапрос списка ТС в ЛК клиента для отображения актуальных данных.
8. При ошибке валидации или сохранения показывать пользователю сообщение с указанием причины ошибки и действия, которое нужно выполнить, без потери введенных данных.

### Бэкенд

1. Реализовать метод получения данных справочника из таблицы `vehicle_types` для заполнения выпадающего списка выбора типа ТС.
2. Реализовать серверный метод создания ТС для текущего клиента.
3. При создании ТС сохранять атрибуты сущности `vehicles`:
   - `client_id`;
   - `license_plate`;
   - `vehicle_type_id`;
   - `brand`;
   - `model`;
   - `color`.
4. Проверять, что:
   - `client_id` соответствует существующей записи в таблице `clients`;
   - `vehicle_type_id` соответствует существующей записи в справочнике `vehicle_types`;
   - ГРЗ не дублирует уже существующее ТС;
   - обязательные поля не пустые: `client_id`, `license_plate`, `vehicle_type_id` (согласно словарю данных, сущность «ТС»).
5. Возвращать на фронтенд созданную запись ТС и данные, необходимые для обновления списка ТС.
6. При ошибке валидации или сохранения бэкенд должен вернуть ошибку для отображения на фронтенде с указанием причины отказа.
