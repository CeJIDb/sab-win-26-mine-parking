# Постановка задачи: Профиль клиента - Форма добавления ТС - Реализовать добавление ТС

Код UC: `UC-3.1`

Use Case: [Реестр Use Case](../../artifacts/use-case/use-case-registry.md), строка `UC-3.1 | Добавить ТС`

Схема БД: [chartdb-postgresql-erd-normalized-public.sql](../../../sql/database/chartdb-postgresql-erd-normalized-public.sql)

## Требование

Добавить ТС.

## Декомпозиция по компонентам архитектуры

### База данных

Создать таблицу:

- `vehicles`

и использовать существующие сущности:

- `vehicle_types`
- `clients`

с атрибутами и связями согласно прилагаемой физической схеме БД.

### Фронтенд

1. Создать макеты для формы добавления ТС в ЛК клиента на экране управления транспортными средствами на основе прототипов и согласовать с аналитиком. В макете предусмотреть поля:
   - ГРЗ;
   - марка;
   - модель;
   - цвет;
   - тип ТС.
2. Сверстать форму добавления ТС по согласованным макетам.
3. При вводе ГРЗ валидировать значение в соответствии со словарем данных сущности `ТС`.
4. Поле выбора типа ТС отобразить в виде выпадающего списка на основе справочника `vehicle_types`.
5. При отправке формы передавать на бэкенд данные нового ТС, привязанные к текущему клиенту.
6. При успешном добавлении ТС обновлять список ТС в ЛК клиента и показывать пользователю результат операции.
7. При ошибке валидации или сохранения показывать пользователю сообщение с указанием причины ошибки и действия, которое нужно выполнить, без потери введенных данных.

### Бэкенд

1. Реализовать серверный метод создания ТС для текущего клиента.
2. При создании ТС сохранять атрибуты сущности `ТС`:
   - `client_id`;
   - `license_plate`;
   - `vehicle_type_id`;
   - `brand`;
   - `model`.
   - `color`.
3. Проверять, что:
   - `vehicle_type_id` соответствует существующей записи в справочнике `vehicle_types`;
   - ГРЗ не дублирует уже существующее ТС.
4. Возвращать на фронтенд созданную запись ТС и данные, необходимые для обновления списка ТС.
5. Предоставить на фронтенд данные справочника `vehicle_types` для заполнения выпадающего списка выбора типа ТС.
6. При ошибке валидации или сохранения бэкенд должен вернуть ошибку для отображения на фронтенде с указанием причины отказа.
