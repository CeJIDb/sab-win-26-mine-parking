# Функциональные требования: Сотрудник

Объект концептуальной модели: **Сотрудник** (учётная запись сотрудника парковки: охранник, управляющий, владелец).  
Источники: `Концептуальная модель с атрибутами.md`, реестр UC (UC-15.1–UC-15.5), User Story Map (роли/права, блокировка учётной записи).

---

## 1. Кандидаты ФТ (Роль.Сотрудник.Операция)

Идентификаторы требований вводятся по схеме `FR-Sotrudnik-XXX`.

### 1.1. Создание и просмотр

| ID ФТ | Имя ФТ | Текст ФТ | Обоснование (кратко) |
| ----- | ------ | -------- | -------------------- |
| FR-Sotrudnik-001 | `Управляющий.Сотрудник.Создать` | Система должна создавать учётную запись Сотрудника (идентификатор, логин, хэш пароля, роль, фамилия, имя, отчество, телефон, email, статус «активен») по инициативе Управляющего или Владельца и сохранять её в хранилище с соблюдением уникальности логина. | Обеспечение заведения учётных записей сотрудников и уникальности логина. |
| FR-Sotrudnik-002 | `Управляющий.Сотрудник.Просмотреть` | Система должна отображать Управляющему или Владельцу данные выбранного Сотрудника: идентификатор, логин, роль, фамилия, имя, отчество, телефон, email, статус. | Обеспечение просмотра полной карточки сотрудника ответственными ролями. |
| FR-Sotrudnik-003 | `Управляющий.Сотрудник.ПросмотретьСписок` | Система должна отображать Управляющему или Владельцу список Сотрудников с атрибутами идентификатор, логин, фамилия, имя, отчество, роль, статус и должна позволять фильтровать список по роли и по статусу (активен / заблокирован). | Обеспечение поиска и обзора сотрудников по роли и статусу. |

### 1.2. Изменение и блокировка

| ID ФТ | Имя ФТ | Текст ФТ | Обоснование (кратко) |
| ----- | ------ | -------- | -------------------- |
| FR-Sotrudnik-004 | `Управляющий.Сотрудник.Изменить` | Система должна позволять Управляющему или Владельцу изменять у Сотрудника атрибуты: фамилия, имя, отчество, телефон, email; при изменении логина система должна проверять уникальность логина в системе. | Обеспечение актуальности контактных данных и уникальности логина при изменении. |

| FR-Sotrudnik-006 | `Управляющий.Сотрудник.Разблокировать` | Система должна переводить выбранного Сотрудника из статуса «заблокирован» в статус «активен» по инициативе Управляющего или Владельца. | Обеспечение восстановления доступа сотрудника при снятии блокировки. |
| FR-Sotrudnik-007 | `Управляющий.Сотрудник.НазначитьРоль` | Система должна изменять атрибут роль Сотрудника (значение из Enum: охранник, управляющий, владелец) по инициативе Управляющего или Владельца; после изменения роли доступ сотрудника должен соответствовать новой роли при следующей авторизации. | Обеспечение корректного разграничения прав доступа по ролям. |

### 1.3. Авторизация и смена пароля

| ID ФТ | Имя ФТ | Текст ФТ | Обоснование (кратко) |
| ----- | ------ | -------- | -------------------- |
| FR-Sotrudnik-008 | `Сотрудник.Сотрудник.Авторизоваться` | Система должна аутентифицировать Сотрудника по паре логин и пароль; при совпадении учётных данных и статусе Сотрудника «активен» система должна предоставлять доступ в интерфейс в соответствии с ролью (админка для управляющего/владельца, интерфейс КПП для охранника); при статусе «заблокирован» или неверных учётных данных система не должна предоставлять доступ и должна сообщать об ошибке входа. | Обеспечение управляемого доступа сотрудников в систему в соответствии с их статусом и ролью. |

## 2. Проверка полноты набора ФТ

- **Покрытие CRUDL по объекту «Сотрудник»**  
  Создать (Создать), Просмотреть (Просмотреть), Изменить (Изменить), ПросмотретьСписок (ПросмотретьСписок) — покрыты. Удаление записи в модели не предусмотрено: жизненный цикл учётки закрывается статусом «заблокирован»; отдельное ФТ на физическое удаление не добавлялось.

- **Покрытие UC-15.x**  
  UC-15.1 (создать профиль) → Управляющий.Сотрудник.Создать; UC-15.2 (просмотреть профиль) → Управляющий.Сотрудник.Просмотреть; UC-15.3 (изменить профиль) → Управляющий.Сотрудник.Изменить и при необходимости Управляющий.Сотрудник.НазначитьРоль; UC-15.4 (просмотреть список профилей) → Управляющий.Сотрудник.ПросмотретьСписок; UC-15.5 (авторизоваться) → Сотрудник.Сотрудник.Авторизоваться. Дополнительно учтены блокировка/разблокировка и смена пароля сотрудником.

- **Управление правами и статусом**  
  Назначение роли (НазначитьРоль), блокировка и разблокировка учётной записи — отражены в отдельных ФТ.

- **Потенциально отсутствующее (рекомендации)**  
  - **Восстановление пароля сотрудника** (например, Управляющий.Сотрудник.СброситьПароль): сброс пароля администратором с выдачей временного пароля или ссылки на смену — при наличии такого сценария добавить отдельное ФТ.  
  - **Журнал действий сотрудников (аудит)**: если требуется просмотр журнала входов или действий по сотрудникам — добавить ФТ вида Управляющий.ЖурналДействийСотрудника.ПросмотретьСписок.  
  - **Интеграция с кадровой/AD/LDAP системой**: при импорте или синхронизации сотрудников из внешней системы — добавить интеграционные ФТ на приём/передачу данных по сотрудникам.

- **Согласованность с концептуальной моделью**  
  Атрибуты и перечисления (роль, статус) приведены в соответствии с разделом «Сотрудник» и Enum в `Концептуальная модель с атрибутами.md`.
