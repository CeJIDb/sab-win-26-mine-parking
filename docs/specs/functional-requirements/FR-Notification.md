## Функциональные требования: объект Уведомление

### Входные UC

- UC-5.1 «Настроить уведомления» (частично через связи с настройками/шаблонами)
- UC-5.2 «Просмотреть уведомление»
- UC-5.3 «Просмотреть список уведомлений»
- UC-5.4 «Просмотреть список уведомлений в адм»
- UC-5.5 «Создать уведомление»
- UC-5.6 «Отправить уведомление»
- UC-5.7 «Просмотреть историю отправленных уведомлений»
- UC-5.8 «Отправить автоматическое уведомление парковочных сессиях»
- UC-5.9 «Отправить автоматическое уведомление договорах»
- UC-5.10 «Отправить автоматическое уведомление оплатах и счетах»

### Кандидаты ФТ (иерархия Роль → Объект.Операция)

#### КлиентФЛ, КлиентЮЛ

- Уведомление.Просмотреть

  Система должна позволять пользователю с ролью КлиентФЛ, КлиентЮЛ просматривать принадлежащий ему объект Уведомление со следующими атрибутами:

  - идентификатор
  - шаблонУведомленияИд
  - клиентИд
  - канал
  - статусДоставки

- Уведомление.ПросмотретьСписок

  Система должна позволять пользователю с ролью КлиентФЛ, КлиентЮЛ просматривать список принадлежащих ему объектов Уведомление с отображением для каждого уведомления следующих атрибутов:

  - идентификатор
  - шаблонУведомленияИд
  - канал
  - статусДоставки.

#### Управляющий, Владелец

- Уведомление.Создать

  Система должна позволять пользователю с ролью Управляющий, Владелец создавать объект Уведомление со следующими атрибутами:

  - идентификатор
  - шаблонУведомленияИд
  - клиентИд
  - канал
  - сотрудникИд

- Уведомление.Отправить

  Система должна позволять пользователю с ролью Управляющий, Владелец инициировать отправку объекта Уведомление по выбранному каналу со следующими атрибутами:

  - идентификатор
  - шаблонУведомленияИд
  - клиентИд
  - канал
  - сотрудникИд

- Уведомление.Просмотреть

  Система должна позволять пользователю с ролью Управляющий, Владелец просматривать объект Уведомление со следующими атрибутами:

  - идентификатор
  - шаблонУведомленияИд
  - клиентИд
  - канал
  - статусДоставки
  - сотрудникИд

- Уведомление.ПросмотретьСписок

  Система должна позволять пользователю с ролью Управляющий, Владелец просматривать список объектов Уведомление с отображением для каждого уведомления следующих атрибутов:

  - идентификатор
  - шаблонУведомленияИд
  - клиентИд
  - канал
  - статусДоставки
  - сотрудникИд

#### СервисУведомлений (смежная система)

- Уведомление.ОбновитьСтатус.СД

  Система должна изменить Уведомление со следующими атрибутами:

  - статусДоставки

- Уведомление.ОбновитьСтатус.СД. статусДоставки

  Система должна изменять [Уведомление]:статусДоставки по следующему алгоритму:

  - при получении успешного ответа от смежной системы доставки для конкретного уведомления [Уведомление]:статусДоставки = «успешно»
  - при получении сообщения об ошибке доставки для конкретного уведомления [Уведомление]:статусДоставки = «ошибка».

#### Система

- Уведомление.ОпределитьКанал.СД. канал

  Система должна изменять [Уведомление]:канал по следующему алгоритму:

  - если для клиента в сущности Настройки уведомлений указано несколько каналов, то система должна выбирать канал в следующем приоритетном порядке: «личный кабинет», затем «e‑mail», затем «SMS»
  - если для клиента доступен только один канал, то система должна устанавливать [Уведомление]:канал равным этому каналу
  - если для клиента не включён ни один канал для соответствующего типа уведомления, система не должна создавать объект Уведомление и должна зафиксировать факт невозможности отправки в журнале аудита.

- Уведомление.Создать.СД. идентификатор

  Система должна изменять [Уведомление]:идентификатор по следующему алгоритму:

  - [Уведомление]:идентификатор = max([Уведомление]:идентификатор) + 1.

### Проверка полноты набора ФТ

- Основные потоки UC-5.2–UC-5.7, UC-5.8–UC-5.10 покрыты:
  - просмотр единичного уведомления и списка уведомлений клиентом
  - создание и отправка уведомлений управляющим/владельцем
  - просмотр истории отправленных уведомлений в админке
  - обновление статуса доставки уведомлений на основе ответов сервиса уведомлений.
- Потенциально отсутствующие ФТ (рекомендации к расширению):
  - более детализированные СД‑ФТ по генерации содержания уведомлений (подстановка плейсхолдеров из связанных объектов Бронирование, Договор, Платеж, Парковочная сессия)
  - ФТ по повторной отправке неуспешных уведомлений и ограничению числа повторных попыток
  - ФТ по фильтрации и поиску уведомлений в списках по типу, каналу, статусу доставки и периоду.
