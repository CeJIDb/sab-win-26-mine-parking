## Критичные расхождения с контекстной диаграммой

- **Отсутствует `Custom Internal Tool` внутри платформы**
  - Сейчас на слайде с границей ответственности (результат 01) внутри платформы перечислены только:
    - Consumer App (ЛК клиента),
    - Admin Panel,
    - интерфейс охранника КПП,
    - «бизнес‑логика»,
    - сайт‑визитка и геосервисы.
  - В `Контекстная диаграмма.md` центральная система включает ещё **`Custom Internal Tool`** (CRM + конструктор договоров + обращения + аналитика и отчётность).
  - **ТЗ для Computer:** добавить `Custom Internal Tool` в список внутренних подсистем платформы (внутренняя зона ответственности), в том же стиле, что Admin Panel и интерфейс КПП.

- **Неверно смоделированы сайт‑визитка и геосервисы**
  - Сейчас `Сайт-визитка и геосервисы` указаны как один блок **внутри платформы**.
  - В артефакте:
    - `Business Site` — внутренняя подсистема;
    - `Геосервисы и Сайт-визитка` описаны как **внешняя система**, получающая данные (информация, тарифы, заполненность).
  - **ТЗ для Computer:**
    - Разделить на два элемента:
      - внутри платформы: `Business Site (информационный сайт)`;
      - снаружи, в блоке интеграций: `Геосервисы и публичный сайт-витрина (получают данные о тарифах и заполненности)` с однонаправленными потоками из платформы.

- **Платёжная система и терминал слиты в одну интеграцию**
  - Сейчас в презентации `Платёжная система онлайн + терминал` фигурируют как одна интеграция.
  - В `Контекстная диаграмма.md` это **две разные внешние системы**:
    - `Платежная система (онлайн)` — свои входные/выходные потоки;
    - `Платежный терминал (офлайн)` — отдельный канал.
  - **ТЗ для Computer:**
    - Разделить блок на два:
      - `Online payment gateway` (статус платежа, id, квитанция, статус возврата);
      - `Offline payment terminal` (статус платежа, id, квитанция).
    - В таблице потоков на слайде «Интеграции и требования» оформить для них **две строки**, опираясь на описания в `Контекстная диаграмма.md`.

- **ЭДО, биллинг/1С и внешняя аналитика объединены в один блок**
  - Сейчас эти три системы показаны как один общий блок интеграций.
  - В артефакте они идут как **три отдельные внешние системы** с разными потоками:
    - `ЭДО` (договора, счета, акты),
    - `Биллинг/1С` (финансовые выгрузки),
    - `Аналитика` (агрегированные данные, отчёты).
  - **ТЗ для Computer:** разделить визуально на три интеграции и, где есть таблица потоков, явно указать отдельные строки для ЭДО, Биллинга и Аналитики.

- **SSO для ФЛ не сопровождается явным ограничением по сотрудникам**
  - В презентации упомянут `SSO / IdP (аутентификация клиентов-ФЛ)`, но не проговорено, что **сотрудники никогда не ходят через SSO**.
  - В артефактах зафиксировано: сотрудники (охранник, управляющий, владелец) всегда входят по логин+пароль+TOTP 2FA, без участия SSO.
  - **ТЗ для Computer:** под блоком SSO добавить короткую подпись (мелким шрифтом):  
    `SSO применяется только для клиентов-ФЛ; сотрудники (охранники, управляющие, владельцы) всегда входят по логину, паролю и TOTP 2FA (без SSO).`

- **Некорректный акцент на договорах ЮЛ в MVP**
  - Сейчас в описании MVP формулировка `ФЛ + ЮЛ (базовый) — договор, оплата, уведомления` создаёт впечатление, что **договоры ЮЛ в MVP уже ведутся внутри платформы**.
  - В `Контекстная диаграмма.md`: для MVP указано, что **договоры ЮЛ заключаются вне системы (ручной режим)**, а полноценный онлайновый документооборот ЮЛ — это уже полная поставка.
  - **ТЗ для Computer:** переформулировать блок MVP так:
    - для ФЛ: договоры и оплаты в системе;
    - для ЮЛ: в MVP — договоры ведутся снаружи, в системе только базовая поддержка учёта и парковочных сценариев.

## Несоответствия/пробелы относительно концептуальной модели

- **Не отражён `Custom Internal Tool` как часть центральной системы**
  - В `Концептуальная модель с атрибутами.md` и `Контекстная диаграмма.md` это важный компонент (CRM, конструктор договоров, обращения, аналитика).
  - В презентации он нигде явно не фигурирует.
  - **ТЗ для Computer:** на слайде с центральной системой и на слайде с концептуальной моделью добавить отдельный блок `Custom Internal Tool` с кратким описанием (CRM + договоры + обращения + отчёты).

- **Аутентификация: SSO и 2FA не привязаны к сущностям `Клиент` и `Сотрудник`**
  - В концептуальной модели:
    - `Клиент` имеет `ssoSubjectId`, `ssoProvider` и `passwordHash`;
    - `Сотрудник` имеет `passwordHash` и `totpSecretEncrypted`.
  - В презентации это только текст про SSO без показа, «где это живёт» в данных.
  - **ТЗ для Computer:** на слайде с концептуальной моделью добавить:
    - около `Клиент`: подпись `Login or SSO (subjectId, provider)`;
    - около `Сотрудник`: подпись `Login + password + TOTP secret`.
