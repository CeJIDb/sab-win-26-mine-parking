## Короткий вывод

Для нашей ИС парковки на этапе учебного проекта лучше выбрать **модульный монолит**, а не микросервисы.

Это значит:

- система развёртывается как **одно основное приложение**;
- внутри приложения есть **отдельные доменные модули** с чёткими границами;
- внешние нестандартные интеграции можно вынести в отдельные адаптеры.

Кто использует систему и через какой интерфейс:

| Роль | Интерфейс | Основные модули |
| --- | --- | --- |
| Клиент ФЛ | Личный кабинет (PWA) | Booking, Payment, Notification |
| Клиент ЮЛ | Личный кабинет | Contracts, Booking, Payment |
| Охранник | АРМ на КПП (контрольно-пропускном пункте) | Access, Session |
| Администратор | Веб-панель | Tariff, Employee, Facility, Report, Support |

## Где здесь DDD

**DDD (Domain-Driven Design, проектирование, ориентированное на предметную область)** помогает разделить систему на части, каждая из которых отвечает за свой кусок предметной области.

В нашем случае это означает:

- `Booking` отвечает за **план** (право на место в определённое время);
- `Session` отвечает за **факт** (машина реально стоит на парковке);
- `Access` отвечает за **решение о допуске** (открывать ли шлагбаум);
- `Tariff` отвечает за **цену** (сколько стоит парковка);
- `Payment` отвечает за **деньги** (оплата, задолженность, чек).

Главная идея DDD здесь такая: **не смешивать разные понятия в одном месте кода**.

Например:

- бронирование — это не парковочная сессия;
- расчёт стоимости — это не оплата;
- договор — это не решение открыть шлагбаум.

Подробнее о том, как устроены bounded contexts (ограниченные контексты) в нашем проекте — см. [DDD Bounded Contexts: учебная версия](ddd-bounded-contexts-study.md).

## Где здесь EDA

**EDA (Event-Driven Architecture, архитектура на основе событий)** удобно использовать там, где модулю не нужен мгновенный ответ, а нужно просто сообщить: «событие произошло».

Примеры событий:

- бронирование создано;
- сессия завершена;
- платёж прошёл;
- доступ был запрещён.

Это полезно, когда:

- другому модулю не нужен немедленный ответ;
- нужно уменьшить зависимость между модулями (например, `Session` при завершении публикует событие «сессия завершена», а `Notification` и `Report` сами решают, что с этим делать — они не зависят от деталей Session);
- действие можно выполнить в фоне.

Такие события удобно передавать в:

- `Notification` — чтобы отправить сообщение клиенту;
- `Report` — чтобы обновить отчёты;
- другие некритичные модули.

Важно: EDA используется **не везде**, а выборочно.

- Для критического пути КПП лучше подходят **синхронные вызовы** — охранник не может ждать цепочку асинхронных обработчиков.
- Для уведомлений, аналитики и фоновых действий хорошо подходят **события**.

## Почему не микросервисы

Микросервисы полезны, когда:

- система очень большая;
- над ней работают несколько независимых команд;
- разные части системы нужно масштабировать отдельно;
- допустима высокая техническая сложность.

Для нашего проекта это пока не так. У нас:

- умеренная нагрузка;
- ограниченные сроки;
- ограниченный бюджет;
- тесно связанные сценарии: `допуск → бронирование → сессия → расчёт → оплата`.

Если сразу выбрать микросервисы, придётся рано решать сложные вопросы:

- как синхронизировать данные между сервисами;
- как делать **распределённые транзакции** (когда одно действие затрагивает несколько независимых сервисов и нужно гарантировать, что либо всё выполнится, либо всё откатится);
- как отлаживать цепочки вызовов через несколько сервисов;
- как поддерживать более сложную инфраструктуру.

Для учебного проекта это даёт слишком много сложности и слишком мало пользы.

## Что выбрано

Мы выбираем **модульный монолит с изолированными адаптерами**.

Основные доменные модули:

- `Access` — принимает решение `allow/deny`;
- `Booking` — управляет бронированиями;
- `Session` — ведёт парковочные сессии;
- `Tariff` — считает стоимость;
- `Payment` — ведёт оплату, задолженность и чеки;
- `Contracts` — работает с договорами;
- `Client` — хранит данные клиента и ТС (транспортного средства);
- `Facility` — хранит структуру парковки;
- `Notification` — готовит уведомления;
- `Support` — ведёт обращения;
- `Employee` — хранит сотрудников и роли;
- `Report` — строит отчёты.

Отдельные адаптеры делятся на два типа.

**Аппаратные адаптеры** — обёртки над физическим оборудованием. Каждый адаптер переводит команды домена в протокол конкретного устройства, и обратно — события от устройства в формат платформы:

| Адаптер | Оборудование |
| --- | --- |
| `LPR/СКУД Adapter` | Камеры LPR (License Plate Recognition) + шлагбаумы СКУД (система контроля и управления доступом) |
| `Дисплей Въезда Adapter` | Дисплей у въездного шлагбаума — показывает приветствие и номер места |
| `Дисплей Выезда Adapter` | Дисплей у выездного шлагбаума — показывает сумму к оплате и статус |
| `Информационное Табло Adapter` | Табло над въездом или в холле — показывает количество свободных мест |
| `Платёжный Терминал Adapter` | Терминал самооплаты у выездного шлагбаума — принимает оплату от клиента |

**Программный адаптер** — фоновый процесс без привязки к физическому устройству:

| Адаптер | Что делает |
| --- | --- |
| `Notification Worker` | Читает очередь уведомлений из БД, отправляет SMS/e-mail через внешние шлюзы |

Общее правило для всех адаптеров: они **не содержат бизнес-логики**. Решение «пускать или нет» принимает доменный контекст `Access`. Расчёт суммы делает `Tariff`. Адаптер только переводит команду или данные и больше ничего не решает.

Подробнее о том, как эти роли выглядят в сценариях въезда и выезда — см. [DDD Bounded Contexts: псевдокод — учебная версия](ddd-pseudocode-study.md).

## Важный архитектурный инвариант

По текущим решениям проекта:

- `Session` не должна существовать без `Booking` — **никогда и ни при каких обстоятельствах**;
- если ТС въезжает без заранее созданной брони, система **обязательно** создаёт `Booking` автоматически;
- создание `Booking` и открытие `Session` должны выполняться **атомарно** (это значит: или оба действия выполняются вместе, или ни одного — промежуточного состояния не бывает).

Зачем это важно: если `Session` откроется, а `Booking` по какой-то причине не создастся, система не будет знать, кому принадлежит место и по какому договору. В этом случае нельзя правильно посчитать стоимость и выставить счёт.

Это жёсткий инвариант из [ADR-002](adr-002-booking-vs-session.md), а не опциональное поведение.

> **Дополнительно:** в учебной версии въезд и выезд координируются синхронно через `Application Service`. После успешной фиксации результата система уже может публиковать события для `Notification` и `Report`.

Координацию сквозных сценариев (авто-бронь + сессия, расчёт + оплата + завершение) выполняет **Application Service** — специальный слой оркестрации, который не принадлежит ни одному доменному контексту, но может вызывать API нескольких.

## Когда решение стоит пересмотреть

К вопросу о микросервисах стоит вернуться, если:

- система вырастет по нагрузке;
- появятся несколько независимых команд;
- отдельные модули потребуют независимых релизов;
- аналитика, платёжный контур или интеграции начнут мешать основному потоку работы КПП.

## Итог

Для учебного проекта модульный монолит — это компромисс между простотой и хорошей архитектурой.

Он позволяет:

- сохранить понятную структуру системы;
- применить DDD через разделение на bounded contexts (ограниченные контексты);
- применять EDA там, где события действительно полезны;
- не перегружать проект сложностью микросервисов слишком рано.
