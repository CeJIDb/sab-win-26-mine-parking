## Context

В сценариях UC-12.1 (въезд) и UC-12.6 (выезд) модуль `Access Control Service` (P4 на DFD L1) принимает решение `allow/deny` за один запрос от СКУД. Чтобы решить, P4 опирается на данные из пяти доменных агрегатов, владельцами которых являются другие модули:

- бронирование по ТС — Booking (D2);
- активная парковочная сессия по ТС — Session (D3);
- ТС и статус клиента (ограничен / черный список) — Client / Fleet (D7);
- задолженность и сумма к оплате — Payment / Billing (D5);
- действующий долгосрочный договор — Contracts (D6).

В [C4 L3](../c4/c4-diagrams.md) этот контур описан как «Access Control сверяется с Booking, Session, Fleet, Contract, Billing». На уровне DFD L1 сверки можно реализовать тремя принципиально разными способами, каждый из которых дает разную картинку потоков данных и разные эксплуатационные характеристики.

Ключевые ограничения:

- [ADR-003](adr-003-modular-monolith.md): модульный монолит на одном развертываемом артефакте с единой PostgreSQL.
- [ADR-001](adr-001-online-access-rights-evaluation.md): право доступа оценивается онлайн, на каждый запрос СКУД, в платформе.
- 600 ММ, до 200 операций в минуту в часы пик, целевое оборудование 4 CPU / 8 GB RAM ([NFR-EXT-PERF-001..003](../../specs/nonfunctional-requirements/nfr-external-quality.md), [CONSTR-HW-004](../../specs/constraints/constr-system.md)).
- Латентность КПП: при физическом шлагбауме задержка свыше 1–1.5 секунды воспринимается как сбой.
- Учебный проект: простота и явность важнее микросервисной чистоты.

## Options

### Option A — RPC к доменным сервисам

P4 обращается к 5 соседним сервисам через публичные интерфейсы: `Booking.findActiveByGRZ()`, `Session.findActiveByGRZ()`, `Fleet.findVehicle()`, `Client.getStatus()`, `Payment.getDebt()`, `Contract.getActive()`. Каждый сервис инкапсулирует свой агрегат и не пускает чужих.

**Плюсы:**

- Строгие границы bounded contexts: P4 не знает чужих схем.
- Естественная подготовка к выделению модулей в микросервисы — границы уже физически зафиксированы интерфейсами.
- Поведение каждого агрегата контролируется его владельцем (бизнес-правила, валидации, кеширование).

**Минусы:**

- 5–7 вызовов на горячем пути. В монолите это method calls (быстро), но цепочка ухудшает читаемость и трассировку.
- Транзакционная сложность: невозможно прочитать снимок согласованного состояния по 5 агрегатам без блокировок.
- Дублирование инфраструктуры: каждый сервис экспонирует read-API для нужд P4.

### Option B — Прямое чтение из общей БД

P4 читает чужие таблицы напрямую через именованные SQL view (`v_access_policy_lookup` или per-aggregate read-views), пишет только в свой агрегат `D1` (`access_logs`, `access_points`).

**Плюсы:**

- Один SQL-запрос с join'ами по 5 таблицам — миллисекунды на принятие решения.
- Снимок согласованного состояния в одной транзакции (`READ COMMITTED` достаточно для оценки прав).
- Минимум инфраструктуры: не нужны read-API у других сервисов.
- Соответствует масштабу учебного проекта и ADR-003.

**Минусы:**

- P4 связан со схемой чужих таблиц — миграция чужой колонки требует пересмотра view.
- Нарушение строгих границ bounded context на уровне физической БД (но не на уровне записи).
- Усложняет будущее выделение модулей в микросервисы — сначала придется заменить SQL на API.

### Option C — CQRS read-model

Владельцы агрегатов публикуют доменные события через outbox; отдельный проектор поддерживает денормализованную таблицу `access_decision_view` (`grz`, `client_status`, `has_active_booking`, `has_active_session`, `debt_amount`, `has_active_contract`). P4 читает только эту таблицу.

**Плюсы:**

- Чистые границы: P4 зависит только от своего read-view.
- Максимальная скорость чтения: один SELECT по primary key.
- Готовность к eventual consistency и распределенной архитектуре.

**Минусы:**

- Требует событийной шины и проектора — неоправданно для одной парковки и одного процесса.
- Eventual consistency: окно рассинхронизации между записью владельца и обновлением проекции делает решения P4 потенциально устаревшими, что плохо для allow/deny на КПП.
- Сложность реализации и отладки.

## Decision

Принять **Option B** — прямое чтение из общей БД через именованные read-only SQL view как основную стратегию для Access Control в платформе парковки.

Ключевые инварианты решения:

1. **Чтение без записи.** P4 имеет право читать чужие таблицы (`bookings`, `parking_sessions`, `vehicles`, `clients`, `debts`, `contracts`). P4 не имеет права в них писать — `INSERT`/`UPDATE`/`DELETE` остается у владельца агрегата. Контролируется на уровне репозитория (только `SELECT`-методы) и прав БД (роль `access_control_reader` без `INSERT/UPDATE/DELETE` на чужие таблицы).
2. **Чтение через именованные view.** Все чужие чтения P4 идут через явные SQL view (`v_access_active_booking_by_grz`, `v_access_active_session_by_grz`, `v_access_client_status`, `v_access_debt_by_session`, `v_access_active_contract_by_client`). View фиксируют контракт чтения и образуют точку, где видно, какие чужие колонки используются — это упрощает миграцию схемы и code review.
3. **Запись только в свой агрегат.** P4 пишет в `D1` (`access_logs`, `access_points`) — журнал решений и конфигурация точек доступа.
4. **На DFD L1 это отображается как direct-read семантика.** Хранилище одно (общая PostgreSQL), стрелки чтения от DB к P4 показаны явно.
5. **Изменение чужой view — breaking change для P4.** Любое переименование колонки или удаление таблицы, которая используется во view P4, требует синхронной правки этих view и регрессионного теста allow/deny.

## Rationale

- **Соответствие ADR-003.** Модульный монолит на единой PostgreSQL прямо допускает прямое чтение чужих таблиц. ADR-003 фиксирует «единая БД для основного процесса» (инвариант 5) и «изоляция по схемам». Инвариант не запрещает чтение между схемами при наличии гранта — он запрещает запись.
- **Соответствие ADR-001.** Онлайн-оценка прав на каждый запрос требует низкой латентности на горячем пути. Прямой SQL быстрее цепочки RPC даже в монолите за счет одной транзакции и одного round-trip к БД.
- **Масштаб не оправдывает CQRS.** До 200 операций в минуту с пиковыми нагрузками на КПП покрываются прямым чтением с подготовленными индексами и view. Eventual consistency на КПП недопустима — клиент не должен получать `deny`, потому что проектор не успел отработать платеж 200 мс назад.
- **Учебный проект.** Введение событийной шины ради чистоты границ Access Control добавит существенную сложность без бизнес-выгоды на текущем масштабе.
- **Согласованность снимка.** Single SQL-транзакция дает консистентный снимок 5 агрегатов; цепочка из 5 RPC такого не гарантирует без распределенных блокировок.

## Trade-offs

- **P4 связан со схемой чужих таблиц.** Митигация — именованные view как контракт чтения; владелец схемы при breaking change обязан синхронно править view P4.
- **Слабее граница между bounded contexts.** Митигация — запрет записи на уровне репозитория и прав БД; код-ревью; статический анализ зависимостей (см. ADR-003 next steps).
- **Сложнее последующая декомпозиция в микросервисы.** Митигация — view являются той же точкой замены: при выделении в сервис view заменяется на API-вызов один-в-один.
- **Риск рассинхронизации логики.** Если в Booking меняется правило «активная бронь», P4 должен использовать ту же логику. Митигация — view определяет «активная бронь» на уровне SQL и является единым источником истины для всех консьюмеров.

## Consequences

### Positive

- Решение allow/deny принимается за один SQL-запрос на КПП — миллисекунды, без распределенных вызовов.
- На DFD L1 контур P4 становится явным: 5 read-стрелок от DB к P4 показывают ровно те агрегаты, на которые опирается политика.
- Снимок согласованного состояния по 5 агрегатам в одной транзакции — корректное поведение в редких race-conditions (одновременная попытка въезда и закрытие сессии).
- Минимум инфраструктуры — не нужны read-API в Booking, Session, Fleet, Client, Payment, Contract.

### Negative

- Связывание P4 со схемой чужих таблиц через view.
- Изменение схемы чужого агрегата требует синхронной правки P4 view и регрессии.
- Неприменимо при выделении P4 в отдельный сервис без дополнительной работы — потребуется заменить view на API-вызовы.

### Mitigation

- **Контракт через view:** все чужие чтения P4 — только через `v_access_*` view; прямые `SELECT` из чужих таблиц в коде P4 запрещены.
- **Права БД:** роль `access_control_reader` имеет `SELECT` на используемые view, не имеет `INSERT/UPDATE/DELETE` на чужие таблицы.
- **Регрессионный тест:** ключевые UC-12.1 / UC-12.6 покрыты интеграционными тестами на полную БД; любая миграция чужой схемы прогоняет их.
- **Документация:** список view P4 поддерживается в [DFD L1](../../artifacts/dfd-l1.md) и в [ERD](../database/erd/erd-normalized-er-model.md) с указанием владельца таблиц.

## Next steps

- **DFD L1:** добавить read-стрелки `DB → P4` с подписями по 5 агрегатам; общая БД оформляется как один узел.
- **ERD / SQL:** определить view `v_access_active_booking_by_grz`, `v_access_active_session_by_grz`, `v_access_client_status`, `v_access_debt_by_session`, `v_access_active_contract_by_client` в схеме Access Control.
- **NFR / SLO:** зафиксировать latency budget allow/deny на КПП (отдельный SLO, не сводимый к NFR-EXT-PERF-002).
- **Ревью при выделении модуля:** при кандидатуре P4 на extraction в сервис — пересмотреть стратегию: view → API.
- **Связь с ADR-003 инв. 5:** не противоречит — записи остаются у владельцев, чтения через view допускаются.
