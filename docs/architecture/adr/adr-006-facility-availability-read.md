## Context

В сценарии бронирования (UC-7.x) и при публикации витрины свободных мест (UC-12.x) `Facility Service` (P12 на DFD L1) отвечает на вопрос «какие парковочные места свободны в интервале времени `[t1, t2]`». На запрос опираются:

- `Booking Service` (P1) — при подборе места под подтверждаемую бронь (поток F40 «свободные места»);
- `Display Adapter` (P18) — для информационных табло КПП (поток F41 «свободные места»);
- `Сайт-визитка` (E15) — для интерактивной карты парковки (поток F37 «свободные места»).

По [ERD](../database/erd/erd-normalized-er-model.md):

- `parking_places` (агрегат P12) хранит только статичные атрибуты места и текущую фактическую занятость `is_occupied BOOLEAN` (записывается P5 при открытии/закрытии сессии);
- резервирование на интервал явно не материализуется в `parking_places`: «текущее резервирование места не хранится в `parking_places`: оно определяется по связанным `bookings` и интервалу времени запроса» (раздел `parking_places`, строка 793 ERD);
- `bookings` (агрегат P1) хранит `parking_place_id` и интервал `[from, to]`.

Чтобы P12 ответил «свободно на интервал», нужно соединить два агрегата: свой `parking_places` (фильтр по `operational_status_id`, типу зоны, разрешенным типам ТС) и чужой `bookings` (исключить места с пересекающимися бронями).

В предыдущей версии DFD L1 ([CHG-20260429-005](../../process/traceability-matrix-log.md)) это было показано как поток F38 «резервирование/освобождение ПМ» от P1 к P12 — командой на флип несуществующего поля `Зарезервировано`. Поток противоречит ERD: писать в `parking_places` это поле некуда, а сама бронь — это запись в `bookings`, владелец которой P1 пишет ее в свой агрегат и без команды P12.

Ключевые ограничения:

- [ADR-003](adr-003-modular-monolith.md) инв. 5: единая PostgreSQL для основного процесса; чтения чужих агрегатов через view допускаются, записи остаются у владельца.
- [ADR-005](adr-005-access-control-direct-db-read.md): паттерн «read через именованные view» уже введен для горячего пути allow/deny на КПП (P4). Решение должно согласовываться по стилю.
- Витрина «свободные места» — не горячий путь allow/deny, но имеет требование к согласованности: после подтверждения брони (write в `bookings`) последующие запросы должны видеть место как занятое.
- На L1 «свободно сейчас» отдает P12 (потоки F41 → P18 и F37 → E15). Это разная по семантике задача от «свободно на интервал»: ей не нужна `bookings`, она работает только с `parking_places.is_occupied + operational_status`. ADR-006 разделяет два контекста и не трогает потоки F37/F41.

## Options

### Option A — RPC P12 → P1 (Booking.findActiveByInterval)

P12 спрашивает у P1 список броней в интервале через публичный интерфейс P1, потом локально соединяет с `parking_places`.

**Плюсы:**

- Строгие границы bounded contexts: P12 не знает схемы `bookings`.
- P1 — естественный владелец логики «активная бронь».

**Минусы:**

- Дополнительный round-trip и дублирование инфраструктуры — P1 экспонирует read-API специально для P12.
- Расходится со стилем ADR-005, в котором аналогичная задача решается через view, а не RPC.
- Снимок согласованного состояния `parking_places` + `bookings` без блокировок одной транзакцией не получить.

### Option B — Прямое чтение P12 через именованный SQL view

P12 читает `bookings` через явный read-only view `v_facility_active_bookings_by_interval` (или `v_facility_availability`, объединяющий `parking_places` и активные брони). Запись в чужой агрегат запрещена.

**Плюсы:**

- Один SQL-запрос с join'ом — миллисекунды для одного парковочного объекта (600 ММ × число активных броней).
- Снимок согласованного состояния в одной транзакции (`READ COMMITTED` достаточно).
- Соответствует уже введенному паттерну ADR-005 — единый стиль чтения чужих агрегатов в монолите.
- Минимум инфраструктуры: P1 не экспонирует отдельное read-API для расчета доступности.

**Минусы:**

- P12 связан со схемой `bookings` через view — миграция чужой колонки требует синхронной правки view.
- Слабее граница bounded context на уровне физической БД (но не записи).

### Option C — P1 считает доступность сам

P1 читает `parking_places` (и связанные справочники зон и типов ТС) через именованный view от P12 и соединяет со своим `bookings` локально. P12 продолжает отдавать «свободно сейчас» для P18 и E15 на собственных данных, но не для P1 — у P1 контекст другой.

**Плюсы:**

- P1 владеет всей логикой подбора места под бронь — статус активной брони, пересечение интервалов, фильтр по типу ТС и льготам.
- P12 остается чистым агрегатором инфраструктуры; не знает про брони.
- Симметрично ADR-005: consumer (P4 для allow/deny, P1 для бронирования) сам собирает данные через свой `v_*` view; владелец агрегата экспонирует только контракт чтения.
- «Свободно сейчас» (без интервала) для P18 и E15 остается у P12 — это другая по семантике задача, и она не страдает от изменения.

**Минусы:**

- P1 связан со схемой `parking_places` и инвентарных справочников через view.
- Требует view над инвентарем со стороны P12 — но это аналог view P4 в ADR-005 и обходится одним SQL-объектом.

## Decision

Принять **Option C** — P1 (Booking Service) сам считает доступность на интервал: соединяет свой агрегат `bookings` с инвентарем мест `parking_places` (и связанными справочниками), который читает через именованный read-only SQL view от P12. Расчет «свободно сейчас» для табло (P18) и сайта-визитки (E15) остается у P12 на его собственных данных (`parking_places.is_occupied + operational_status`) и в ADR-006 не меняется.

> Решение пересмотрено по результату ревью: исходно был выбран Option B (P12 читает `bookings`), но он размывал роль P12 как агрегатора инфраструктуры и нес бизнес-логику бронирования (определение «активная бронь», пересечение интервалов) в чужой контекст. Option C отделяет два разных вопроса с разной семантикой: «свободно сейчас» (без интервала, у P12) и «свободно на интервал» (с пересечением броней, у P1).

Ключевые инварианты решения:

1. **Расчет доступности на интервал — у P1.** Логика «активная бронь», пересечение интервалов `[t1, t2]`, фильтр по типу ТС, льготам и эксплуатационному статусу выполняется в Booking Service. P1 читает свой `bookings` напрямую и инвентарь мест P12 — через view.
2. **Чтение инвентаря через именованный view.** P1 имеет право читать `parking_places` и связанные справочники (`sectors`, `zone_types`, `zone_type_vehicle_types`, `operational_statuses`, опционально override-`tariffs`) только через явный SQL view (рабочее имя `v_booking_inventory`). Прямые `SELECT` из чужих таблиц P12 в коде P1 запрещены.
3. **Запись только в свой агрегат.** P1 пишет в `bookings`, `booking_status_history`, `outbox_events` — это уже зафиксировано (F11, F17). В чужие таблицы P1 не пишет: `INSERT/UPDATE/DELETE` на `parking_places` и инвентарь — только у P12.
4. **`is_occupied` остается реальной колонкой и обновляется P12 по команде P5.** Текущая занятость «сейчас» в `parking_places.is_occupied` пишется только P12 (поток F39 «занятие/освобождение ПМ»). Бронь на интервал в `parking_places` не материализуется.
5. **Команды резервирования от P1 нет.** Поток F38 «резервирование/освобождение ПМ» удаляется из DFD L1 — он не имеет данных под собой и противоречит ERD. Бронь = запись в `bookings`, и она показана как F11 (P1 → DB).
6. **«Свободно сейчас» для P18 и E15 — у P12 на собственных данных.** Потоки F41 (P12 → P18 «свободные места») и F37 (P12 → E15 «свободные места») остаются за P12; ему для этого `bookings` не нужен.
7. **Изменение схемы инвентаря — breaking change для P1.** Любое переименование колонки или удаление справочника, используемого во `v_booking_inventory`, требует синхронной правки view P12 и регрессионного теста подбора места при бронировании.

## Rationale

- **Согласованность с ADR-005.** В ADR-005 P4 сам собирает у себя политику из чужих агрегатов через `v_access_*` view, а не делегирует расчет владельцам. Симметрично P1 сам собирает доступность из своего `bookings` и чужого `parking_places` через `v_booking_inventory` — оба сервиса остаются consumer'ами view, а не делегируют логику владельцам.
- **Согласованность с ADR-003 инв. 5.** Записи остаются у владельца (P12 пишет инвентарь, P1 пишет брони), чтения через view допускаются — это и есть закрепленный инвариант.
- **Соответствие ERD.** Раздел `parking_places` явно говорит, что «текущее резервирование места не хранится в `parking_places`: оно определяется по связанным `bookings` и интервалу времени запроса». ADR-006 закрепляет, что определение это делает Booking — владелец `bookings` и понятия «бронь» как такового.
- **Чистые границы bounded contexts.** Бизнес-правила Booking (статус активной брони, пересечение интервалов, правила подбора под тип ТС и льготы) живут в Booking. P12 владеет инфраструктурой, не знает про брони — это упрощает его и не дублирует семантику «активная бронь» с view P4.
- **Разные вопросы — разные ответы.** «Свободно сейчас» (P18, E15) и «свободно на интервал» (P1) — две разные доменные операции. Совмещать их в одном агрегаторе нет смысла: бронь как понятие нужна только Booking.
- **Учебный проект.** Прямое чтение через view проще RPC и не требует событийной шины; это не противоречит SLA расчета доступности (не горячий путь КПП).

## Trade-offs

- **P1 связан со схемой инвентаря P12.** Митигация — именованный view как контракт чтения; владелец инвентаря (P12) при breaking change обязан синхронно править `v_booking_inventory`.
- **Слабее граница между bounded contexts (Booking ↔ Facility).** Митигация — запрет записи на уровне репозитория и прав БД; код-ревью.
- **Сложнее последующая декомпозиция в микросервисы.** Митигация — view является точкой замены: при выделении P12 в сервис `v_booking_inventory` меняется на API-вызов P12 один-в-один.
- **Если в будущем потребуется единая «витрина доступности» для нескольких consumer'ов** — нужно будет либо вводить отдельный модуль/проектор, либо возвращаться к Option B. На текущем масштабе единственный consumer на интервальный расчет — P1.

## Consequences

### Positive

- Один SQL-запрос P1 с join'ом своего `bookings` и view `v_booking_inventory` дает консистентную картину доступности на интервал.
- На DFD L1 контур чтения становится симметричным с ADR-005: read-стрелка `DB → P1` показывает чтение инвентаря через view; команда P1 → P12 «резервирование» исчезает как лишняя; F40 «свободные места» (P12 → P1) тоже исчезает.
- P12 не знает про брони — его роль агрегатора инфраструктуры остается чистой; для P18 и E15 он отдает «свободно сейчас» на собственных данных.
- Минимум инфраструктуры — ни P1, ни P12 не экспонируют отдельные read-API друг для друга.
- Единый стиль с ADR-005: оба чтения чужих агрегатов в монолите идут через `v_*` view, owned by consumer.

### Negative

- Связывание P1 со схемой инвентаря P12 через view.
- Изменение схемы `parking_places`/`sectors`/`zone_types` требует синхронной правки `v_booking_inventory`.
- Неприменимо при выделении P12 в отдельный сервис без дополнительной работы — потребуется заменить view на API-вызов P12.

### Mitigation

- **Контракт через view:** все чтения инвентаря из P1 — только через `v_booking_inventory`; прямые `SELECT` из `parking_places` и связанных справочников в коде P1 запрещены.
- **Права БД:** роль `booking_reader` имеет `SELECT` на `v_booking_inventory`, не имеет `INSERT/UPDATE/DELETE` на инвентарные таблицы.
- **Регрессионный тест:** подбор места при бронировании покрыт интеграционным тестом; миграция инвентаря прогоняет тесты P1 и P12.
- **Документация:** список view P1 поддерживается в [DFD L1](../../artifacts/dfd-l1.md) и в [ERD](../database/erd/erd-normalized-er-model.md) с указанием владельца таблиц.

## Next steps

- **DFD L1:** удалить F38 (P1 → P12 «резервирование/освобождение ПМ») и F40 (P12 → P1 «свободные места»); добавить read-стрелку `DB → P1` для чтения инвентаря через view `v_booking_inventory`; F41 (P12 → P18) и F37 (P12 → E15) остаются без изменений.
- **ERD / SQL:** определить view `v_booking_inventory` в схеме Facility (или общая read-схема); включить `parking_places`, `sectors`, `zone_types`, `zone_type_vehicle_types`, `operational_statuses`, опционально override-`tariffs`; запись через view запрещена на уровне грантов.
- **Связь с ADR-003 инв. 5:** не противоречит — записи остаются у владельца (P12), чтения через view допускаются.
- **Связь с ADR-005:** одинаковый паттерн `consumer-owned read view`; при пересмотре одного из ADR — учесть второе.
- **Ревью при выделении модуля:** при кандидатуре P12 на extraction в сервис — пересмотреть стратегию: view → API-вызов P12.
