# Регламент взаимодействия ИС

Документ фиксирует регламент межсистемного взаимодействия цифровой платформы парковки: кто кому что передает, с какой периодичностью и по какому поводу.

Регламент сводит в одну таблицу направления обмена, состав передаваемых данных и событийную модель. Он используется как опорная таблица для интеграционных сценариев, UML Sequence и контрактов обмена.

## Назначение

Регламент нужен, чтобы:

- зафиксировать перечень межсистемных обменов в одном месте;
- согласовать направление обмена, состав передаваемых данных и триггер передачи;
- дать общую рамку для дальнейшей детализации в UML Sequence, маппингах и API-контрактах.

Регламент описывает направления обмена на уровне «кто кому что передает и по какому событию». Конкретные схемы сообщений и полная семантика полей задаются в контрактах обмена и маппингах (см. раздел [Связанные документы](#связанные-документы)).

## Границы регламента

Включено:

- взаимодействие СКУД, адаптера СКУД и сервиса контроля доступа;
- взаимодействие ЛК клиента, платформы парковки, платежного сервиса, адаптера платежей и внешнего платежного провайдера в сценарии онлайн-оплаты парковочной сессии.

Не включено:

- обмен с ОФД и фискальные данные чека;
- протоколы физических устройств на объекте (шлагбаумы, терминалы);
- UI-детали редиректа и экраны оплаты у провайдера.

## Условные обозначения

- **Полный объем / изменения** — передается целиком или только дельта относительно предыдущего состояния.
- **Периодичность обмена / событие** — что именно запускает передачу.
- **Протокол взаимодействия** — транспорт и формат обмена (уточняется в контрактах API).
- **Полезная нагрузка** — пример или схема сообщения (уточняется в соответствующих артефактах JSON/XML).

## Блок 2. Онлайн-оплата парковочной сессии

Описывает направления обмена между ЛК / PWA клиента, платформой парковки, платежным сервисом, адаптером платежей и внешним платежным провайдером.

| Система-источник                    | Система-приемник                  | Передаваемые данные                                                                                                                                                                                                                            | Полный объем / изменения | Периодичность обмена / событие                                 | Протокол взаимодействия | Полезная нагрузка |
| ----------------------------------- | --------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------ | -------------------------------------------------------------- | ----------------------- | ----------------- |
| ЛК / PWA                            | Платформа парковки (Система), API | Запрос подготовки онлайн-оплаты: идентификатор клиента, идентификатор парковочной сессии, идентификатор бронирования, способ оплаты (`онлайн`), при необходимости идентификатор корреляции запроса                                             | Полный                   | По событию: команда клиента «Оплатить онлайн» в ЛК             | уточняется              | уточняется        |
| Платежный сервис → Адаптер платежей | Платежный провайдер               | Запрос регистрации операции оплаты: `сумма`, `валюта`, `способОплаты`, `внутреннийИдПлатежа`, `назначениеПлатежа`, `срокОплаты` (при необходимости), `URL возврата`, реквизиты идемпотентности и сверки (в т.ч. сопоставимые с `идПровайдера`) | Полный                   | По событию: после успешной проверки основания оплаты в Систему | уточняется              | уточняется        |
| Адаптер платежей → Платежный сервис | ЛК / браузер клиента              | `redirect_url` или данные для открытия страницы оплаты у провайдера                                                                                                                                                                            | Полный                   | По событию: после успешной регистрации операции у провайдера   | уточняется              | уточняется        |

Последовательность вызовов показана в [UML Sequence — UC-10.2 Онлайн-оплата краткосрочной аренды](sequence-uc-10-2-pay-online-short-term-rental.md). ## Открытые вопросы

- Протокол взаимодействия для обменов блока СКУД (REST, MQTT, проприетарный SDK вендора) — требует отдельного архитектурного решения.
- Состав реквизитов идемпотентности и сверки для провайдера платежей — уточняется в контракте интеграции с конкретным провайдером.
- Полезная нагрузка обменов СКУД — требует согласования с вендором оборудования.
