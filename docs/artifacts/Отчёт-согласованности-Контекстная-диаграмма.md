# Отчёт о согласованности: Контекстная диаграмма (полная поставка)

## 1. Резюме

**Контекстная диаграмма полной поставки в целом согласована** с Impact Map, Opportunity Canvas, User Story Map, Картой навигации, Схемой парковки AS-IS и Карточкой проекта по акторам, внешним системам, типам зон, параметрам ТС, льготным категориям и терминологии. Потоки данных «в систему» / «из системы» соответствуют описанным сценариям и ролям. Таблица отличий полной поставки от MVP в конце документа соответствует остальным артефактам.

**Имеются частичные расхождения и пробелы:**

- ~~В Карточке проекта перечень смежных систем короче~~ — **исправлено:** в Карточку добавлены ЭДО, биллинг/1С и система аналитики с пометкой «полная поставка» и ссылкой на контекстную диаграмму.
- Протоколы: сверка формулировок контекстной диаграммы с текстом протоколов выполнена (акторы, границы системы, СКУД/LPR, ФЛ без ЛК, терминология). Расхождение по нумерации было только в Карте навигации (пропущены №1.1 и №2) — исправлено.
- Незначительное уточнение: в СКУД в полной диаграмме указан поток «при наличии: подтверждение открытия/закрытия шлагбаума, статус устройства» — в эталоне MVP.txt такого потока нет; расхождение допустимо как расширение для полной поставки.

Рекомендации: нумерация протоколов унифицирована в Карте навигации на «протоколам №1.1–№5».

### 2.4. Терминология

- **ГРЗ, ТС, ПМ** — по глоссарию протокола №4 и User Story Map; в Карточке проекта и на диаграмме совпадают.
- **Тариф, бронирование, договор** — по глоссарию протокола №4 (User Story Map); в диаграмме и Карточке согласованы.
- **ОВЗ, типы зон** — по схеме AS-IS и Карточке; в диаграмме использованы единообразно.

### 2.5. Потоки данных и сценарии

- **ЮЛ:** профиль, реквизиты, банковские реквизиты, ГРЗ и список ТС (добавление/изменение/удаление), параметры ТС, ПМ и привязка к договору, тип зоны, договор, оплата, парковочная сессия, параметры бронирования (дата, время, зона, место), согласия, обращения, скан документов — соответствуют Impact Map (самостоятельное управление списком ТС, договоры, оплата, бронирование, обращения) и User Story Map (блоки 6–10).
- **ФЛ:** профиль, льготная категория, ГРЗ и параметры ТС, данные и параметры договора (для ФЛ по договору), оплата, бронирование, минимальная идентификация при гостевом проезде (телефон, имя, согласие с тарифом) — соответствуют протоколу №5 (минимальная идентификация, ФЛ без ЛК через табло/терминал) и User Story Map.
- **Оператор КПП:** ручной ввод ГРЗ, решение по доступу, поиск профиля, парковочная сессия (создание/корректировка), обработка исключений (чёрный список, отказ от оплаты), инициация оплаты через терминал — соответствуют Impact Map (интерфейс поддержки офлайн-клиентов), Opportunity Canvas (резервная идентификация, исключения).
- **Администратор:** профили (клиенты, охранники, управляющие), договоры и шаблоны, тарифы по типу зоны/ТС/льготам, территория и зоны с типом (по AS-IS), обращения, запросы отчётности; из системы — аналитика, дашборды, выгрузки — соответствуют Impact Map (админ-панель, CRM, тарифы, шаблоны договоров, управление пользователями) и Карте навигации.

### 2.6. Центральная система (границы)

- **Business Site, Consumer App, Admin Panel, Custom Internal Tool, интерфейс оператора КПП** — соответствуют Протоколу №4 (по упоминаниям в диаграмме и в отчёте по Карточке), Карте навигации (сайт-визитка, ЛК клиента, ЛК админа, рабочий интерфейс охранника).
