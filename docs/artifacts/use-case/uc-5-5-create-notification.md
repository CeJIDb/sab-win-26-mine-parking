## Детализация содержания

### UC-5.5 «Создать уведомление»

**Действующее лицо:** Управляющий, Владелец, Система, Севрис уведомлений

**Предусловия:**

- Сотрудник (управляющий или владелец) авторизован в административной панели.
- Шаблон уведомления создан и опубликован (UC-6.1).
- Список получателей определён: один или несколько клиентов (ФЛ и/или ЮЛ), либо сегмент
  клиентов.
- Сервис уведомлений (SMS/Email-шлюз) доступен и интегрирован с системой.
- Модуль уведомлений функционирует и доступен для записи.
- У каждого получателя в профиле зафиксированы контактные данные (телефон и/или
  e-mail).

**Триггер:**

Сотрудник в административной панели переходит в раздел "Уведомления" и нажимает кнопку
"Создать уведомление" → "Подтвердить".

**Постусловие:**

- Для каждой пары "получатель-канал" в системе существует запись уведомления с
  заполненными клиентИд, шаблонУведомленияИд, канал, сотрудникИд.
- Записи уведомлений, по которым Сервис уведомлений принял запрос к отправке, имеют
  статусДоставки = успешно; записи, по которым запрос отклонён или не получен ответ, -
  статусДоставки = ошибка. Фактическая доставка до устройства получателя системой не
  отслеживается.
- Внутрисистемный Push сохранён в ЛК каждого получателя со статусом "Не прочитано".
- По каждому сконфигурированному внешнему каналу (SMS/e-mail) отправлен запрос в
  Сервис уведомлений.
- История отправленных уведомлений обновлена (доступна сотруднику в разделе UC-5.6).
- Сотрудник видит итоговый статус операции: количество записей со статусДоставки =
  успешно и со статусДоставки = ошибка.

**Основной поток:**

1. Сотрудник выбирает получателей уведомления:

- один конкретный клиент (поиск по ФИО, ГРЗ, номеру телефона);
- несколько клиентов (мультивыбор из списка);
- сегмент клиентов (по фильтрам: тип клиента, статус договора, задолженность и т.д.).

2. Сотрудник выбирает шаблон уведомления из числа опубликованных (UC-6.2). Выбор
   шаблона обязателен.
3. Сотрудник выбирает каналы доставки:

- внутрисистемный Push (обязательный канал, включён по умолчанию);
- SMS (если у получателей есть телефон и включён данный канал);
- e-mail (если у получателей есть e-mail и включён данный канал).

4. Система выполняет предварительную валидацию:

- проверяет наличие хотя бы одного получателя;
- проверяет, что шаблон уведомления выбран;
- проверяет наличие соответствующих контактных данных у получателей для выбранных
  каналов.

5. Система создаёт отдельную запись уведомления для каждой пары "получатель-канал" со следующими атрибутами:

- клиентИд = [ID клиента-получателя]
- шаблонУведомленияИд = [ID выбранного шаблона]
- канал = [один из: личный кабинет, SMS, e-mail]
- статусДоставки = ошибка (начальное значение до получения подтверждения)
- сотрудникИд = [ID сотрудника, создавшего уведомление]
- датаСоздания = текущая дата и время

6. Система передаёт команду в Модуль уведомлений для сохранения Push в ЛК каждого
   получателя:

- Push сохраняется со статусом "Не прочитано";
- получатель увидит Push при следующем открытии приложения/ЛК.

7. Для каждого получателя и каждого внешнего канала (SMS/e-mail) Система передаёт запрос
   в Сервис уведомлений:

- получатель = [контактные данные клиента]
- текст = [содержимое уведомления]
- канал = SMS или e-mail
- идентификаторУведомления = [ID записи уведомления]

8. Система получает от Сервиса уведомлений подтверждение о принятии запроса к отправке.
9. Система обновляет уведомление для каждой принятой записи:

- статусДоставки = успешно (означает, что Сервис уведомлений принял запрос к
  отправке; фактическая доставка до устройства получателя не гарантируется и системой
  не отслеживается)

10. Система обновляет историю отправленных уведомлений (UC-5.6) и отображает
    сотруднику итоговый статус: количество получателей и каналов, по которым отправка
    выполнена успешно.
11. Сценарий завершается.
