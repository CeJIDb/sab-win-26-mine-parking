## Предусловия

- Клиент ЮЛ авторизован в ЛК.
- В профиле клиента заполнены данные организации (UC-2.13 «Добавить данные организации»).
- В профиле клиента заполнены банковские реквизиты организации (хотя бы один счет, отмеченный как основной).
- В профиле клиента есть хотя бы одно ТС.
- Клиент подключен к одному из операторов ЭДО, поддерживаемых платформой (или есть роуминг с тем оператором, где работает Владелец).
- У Владельца есть аккаунт и действующая КЭП в том же операторе ЭДО (или совместимом через роуминг) для подписания договоров со стороны платформы.
- Существует опубликованный шаблон договора долгосрочной аренды для ЮЛ.
- Доступны для бронирования машиноместа в нужном секторе на нужный срок.

## Триггер

Клиент ЮЛ в ЛК переходит в раздел «Создать договор».

## Постусловие

- `[договор].статус` = `активен`.
- `[договор].файлДоговора` указывает на сохраненный финальный документ с двумя ЭП.
- `[договор].датаПодписанияКлиентом` и `[договор].датаПодписанияПарковкой` заполнены.
- На указанные клиентом машиноместа созданы бронирования на весь срок договора.
- Клиент получил уведомление об успешном заключении договора и доступности договора в ЛК.
- Управляющий получил уведомление о новом подписанном договоре.

## Основной поток

1. Клиент в форме «Создать договор» указывает параметры аренды: срок (`датаНачала`, `датаОкончания`), тариф из доступных, ТС из профиля, машиноместа.
2. Система рассчитывает стоимость по выбранному клиентом тарифу и отображает условия договора (сумма, период, состав машиномест, ТС).
3. Клиент подтверждает «Заключить договор» → [Ok].
4. Система создает `[договор]` со следующими атрибутами:
   - `идентификатор` — автоген;
   - `клиентИд` — ссылка на клиента из сессии;
   - `шаблонДоговораИд` — ссылка на опубликованный шаблон долгосрочной аренды для ЮЛ;
   - `номерДоговора` — автоген по правилу нумерации;
   - `датаНачала`, `датаОкончания` — из формы;
   - `статус` = `черновик`;
   - `файлДоговора` = `null`.
5. Система формирует документ договора по шаблону договора с подстановкой реквизитов ЮЛ из профиля клиента и параметров из формы.
6. Система передает документ в ЭДО с маршрутом согласования: клиент → Владелец.
7. Система получает от ЭДО подтверждение приема документа с идентификатором документа в ЭДО.
8. Система обновляет `[договор]`: `статус` = `на подписании клиентом`.
9. Система отправляет клиенту уведомление «договор отправлен на подпись в ЭДО».
10. Система получает от ЭДО событие «документ подписан клиентом».
11. Система обновляет `[договор]`: `статус` = `на подписании парковкой`, `датаПодписанияКлиентом` = момент события из ЭДО.
12. Система получает от ЭДО документ, подписанный обеими сторонами (с двумя ЭП).
13. Система сохраняет финальный документ во внешнем хранилище.
14. Система обновляет `[договор]`: `статус` = `активен`, `файлДоговора` = ссылка на сохраненный документ, `датаПодписанияПарковкой` = момент получения финального документа.
15. Система создает бронирования на указанные клиентом машиноместа на весь срок договора.
16. Система отправляет клиенту уведомление об успешном заключении договора и доступности договора в ЛК.
17. Система отправляет управляющему уведомление о новом подписанном договоре.
18. Сценарий завершается.

## Расширение

**3а. У клиента уже есть договор с тем же шаблоном и пересекающимся составом машиномест за пересекающийся период**

3а1. Система определяет существующий `[договор]` по бизнес-ключу: тот же `клиентИд`, тот же `шаблонДоговораИд`, пересекающийся состав машиномест, пересекающийся период `датаНачала`–`датаОкончания`, статус не финальный (`черновик`, `на согласовании`, `на подписании клиентом` или `на подписании парковкой`).
3а2. Система не создает новый `[договор]`, возвращает клиенту существующий с его текущим статусом.
3а3. Сценарий завершается.

**6а. ЭДО недоступно при отправке документа**

6а1. Система фиксирует ошибку отправки в ЭДО (сетевая ошибка / таймаут запроса).
6а2. `[договор]` остается в статусе `черновик`, `файлДоговора` = `null`.
6а3. Система отправляет клиенту уведомление об ошибке отправки документа в ЭДО.
6а4. Сценарий прерывается.

**7а. Подтверждение приема от ЭДО не получено**

7а1. Система фиксирует отсутствие подтверждения приема (таймаут ответа или ответ без идентификатора документа).
7а2. `[договор]` остается в статусе `черновик`, `файлДоговора` = `null`.
7а3. Система отправляет клиенту уведомление об ошибке отправки документа в ЭДО.
7а4. Сценарий прерывается.

**10а. Из ЭДО получено событие отказа клиента от подписания**

10а1. Система получает от ЭДО событие отказа клиента от подписания.
10а2. Система обновляет `[договор]`: `статус` = `отклонен клиентом`.
10а3. Система отправляет клиенту уведомление об отказе от подписания.
10а4. Сценарий завершается.

**10б. Событие подписания клиентом не получено в течение SLA**

10б1. Система фиксирует таймаут ожидания подписи клиента (SLA = 7 календарных дней с момента отправки документа в ЭДО).
10б2. Система обновляет `[договор]`: `статус` = `просрочен`.
10б3. Система отправляет клиенту уведомление об истечении срока подписания.
10б4. Сценарий завершается.

**12а. Из ЭДО получено событие отказа Владельца от подписания**

12а1. Система получает от ЭДО событие отказа Владельца от подписания.
12а2. Система обновляет `[договор]`: `статус` = `отклонен парковкой`.
12а3. Система отправляет клиенту уведомление об отказе со стороны парковки.
12а4. Сценарий завершается.

**12б. Финальный документ не получен в течение SLA**

12б1. Система фиксирует таймаут ожидания финального документа с двумя ЭП (SLA = 7 календарных дней с момента подписи клиента).
12б2. Система обновляет `[договор]`: `статус` = `просрочен`.
12б3. Система отправляет клиенту уведомление об истечении срока подписания со стороны парковки.
12б4. Сценарий завершается.
