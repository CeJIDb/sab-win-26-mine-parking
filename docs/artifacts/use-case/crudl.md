## Назначение

Артефакт фиксирует каноничную CRUDL-матрицу проекта: какие роли и внешние системы создают, просматривают, изменяют, удаляют и перечисляют ключевые сущности предметной области парковки.

## Контекст и источник

- Этап проекта: Этап 2. Моделирование взаимодействия и разработка пользовательских требований
- Тип артефакта: CRUDL-матрица
- Источник: импортированный проектный материал по сценариям использования, реестр use case, рабочее моделирование команды
- Статус: каноничное описание CRUDL для use case и функциональных требований

## Диаграмма

В текущей версии отдельное изображение не приложено; артефакт оформлен как текстовая матрица.

## Текстовое описание

CRUDL-матрица показывает ролевое покрытие операций над сущностями системы парковки. Она нужна как мост между реестром use case, концептуальной моделью и функциональными требованиями: помогает увидеть, где используется классический жизненный цикл данных, какие операции доступны клиентам и сотрудникам, а какие действия выполняются системами автоматически или через внешние интеграции.

## Ключевые элементы

- Роли пользователей: клиент ФЛ, клиент ЮЛ, владелец, управляющий, охранник или оператор
- Системные участники: система парковки, система уведомлений, СКУД, платежная система, платежный терминал
- Сущности предметной области: профиль клиента, бронирование, парковочная сессия, договор, платеж, тариф, сектор, парковочное место, профиль сотрудника и другие
- Операции CRUDL: `C`, `R`, `U`, `D`, `L`

## Логика артефакта

Матрица читается по строкам: для каждой роли или системы показан набор допустимых операций над сущностями. Такой формат полезен при декомпозиции use case в FR и при сверке с моделью данных: стандартные действия `Создать`, `Просмотреть`, `Изменить`, `Удалить`, `Просмотреть список` должны опираться на эту таблицу, а нестандартные доменные сценарии вроде оплаты, идентификации или завершения сессии уточняются уже в самих UC.

### Легенда

| Буква | Значение |
| --- | --- |
| `C` | create (`создать`) |
| `R` | read (`прочитать` / `просмотреть`) |
| `U` | update (`обновить` / `изменить`) |
| `D` | delete (`удалить`) |
| `L` | list (`просмотреть список`) |

## Выводы и решения

- Матрица задает каноничную опору для стандартных CRUDL-действий в FR и проверке покрытия use case.
- Наиболее насыщенные сущности по набору ролей: профиль клиента, ТС, бронирование, тарифы, сектор и парковочное место.
- Часть операций выполняется не пользователем, а системой или внешней интеграцией, что важно отделять от пользовательских сценариев.

## Ограничения и открытые вопросы

- Матрица не заменяет реестр use case и не описывает последовательность шагов, условия и бизнес-правила сценариев.
- Для нестандартных доменных действий следует ориентироваться на названия UC, а не пытаться искусственно свести их к CRUDL.
- При изменении состава сущностей, ролей или системных интеграций таблицу нужно синхронизировать с реестром use case, моделью данных и FR.
