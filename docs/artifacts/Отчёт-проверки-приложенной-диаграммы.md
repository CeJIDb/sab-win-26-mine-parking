# Отчёт о проверке приложенной контекстной диаграммы на соответствие проекту

**Дата проверки:** 23.02.2026  
**Проверяемый артефакт:** контекстная диаграмма, приложенная к запросу (изображение)  
**Эталон проекта:** `docs/artifacts/Контекстная диаграмма.md`, протоколы №1.1–№5, Impact Map, User Story Map, Карточка проекта

---

## 1. Вывод

**Приложенная контекстная диаграмма не соответствует файлам проекта.** Она описывает другую информационную систему — **«Информационная система Перевозки»** (домен: переводы/перевозки), тогда как проект SAB WIN'26 — это **цифровая платформа парковок** (домен: парковка, КПП, ГРЗ, доступ, тарифы по зонам). Центральная система, акторы, внешние системы и потоки данных в приложенной диаграмме не совпадают с артефактами проекта.

---

## 2. Сравнение по ключевым элементам

| Элемент | Приложенная диаграмма | Проект (Контекстная диаграмма.md и др.) |
|--------|------------------------|------------------------------------------|
| **Центральная система** | Информационная система **Перевозки** | **Цифровая платформа парковок** |
| **Домен** | Переводы, данные о переводах, отчёты по переводам, верификация, контроль рисков | Парковка: въезд/выезд, ГРЗ, парковочные сессии, договоры, тарифы по типам зон, бронирование, доступ |
| **Акторы (люди)** | ФЛ (физическое лицо) | **ЮЛ**, **ФЛ**, **Оператор КПП** (охранник/старший охранник), **Администратор** (управляющий/владелец) |
| **Внешние системы** | СДО (система дистанционного обслуживания), **СС (SSO)**, Гос. органы, СУБД, Обогащение данных, Заказчик проекта, СКД, Бухгалтерия, СБ, Система контроля рисков, Платежный шлюз, Служба уведомлений | Платежная система (онлайн), **Платежный терминал (офлайн)**, ОФД, **СКУД с LPR-камерами**, Сервис уведомлений, **SSO (IdP)** — только для ФЛ, Геосервисы и сайт-визитка, ЭДО, Биллинг/1С, Система аналитики; оборудование: **табло КПП**, **дисплеи въезд/выезд** |

| **SSO** | СС: запрос на авторизацию, данные пользователя (система → SSO); результат авторизации, данные авторизации (SSO → система). Упоминается передача «данные пользователя» в SSO (проvisioning). | SSO/IdP: только для входа **ФЛ** в ЛК. В систему: результат аутентификации, subject id, атрибуты (телефон, e-mail, ФИО). Из системы: запрос на аутентификацию, параметры (client_id, scope и т.п.). Без provisioning в SSO со стороны платформы. |

## 3. Чего нет в приложенной диаграмме (но есть в проекте)

- **ЮЛ** (юридическое лицо) как отдельный актор  
- **Оператор КПП** (охранник, старший охранник)  
- **Администратор** (управляющий, владелец)  
- **СКУД с LPR** (распознавание ГРЗ, право доступа)  
- **Платежный терминал** (офлайн на КПП/выезде)  
- **ОФД** (фискализация)  
- **Табло и дисплеи** на въезде/выезде  
- Понятия: **ГРЗ**, **ПМ**, **типы зон**, **парковочная сессия**, **бронирование**, **договор** (в смысле парковки), **льготные категории**, **ФЛ без ЛК** (гостевой проезд через табло/терминал/персонал)

---

## 4. Что есть в приложенной диаграмме, но не в проекте

- **Информационная система Перевозки** (центральная система)  
- **СДО** (система дистанционного обслуживания)  
- **Гос. органы**, **СУБД**, **Обогащение данных**, **Заказчик проекта**, **СКД**, **Бухгалтерия**, **СБ**, **Система контроля рисков**, **Модуль верификации**, **Модуль контроля рисков**  
- Потоки: заявка на регистрацию, данные по переводу, данные для отчёта (в контексте переводов), результат верификации, инструкции по регистрации/авторизации ФЛ (от СДО), передача данных пользователя в SSO (проvisioning)

Приложенная диаграмма относится к системе «Перевозки» и для проекта парковок не релевантна.
