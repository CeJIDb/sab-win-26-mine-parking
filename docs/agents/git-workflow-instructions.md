# Инструкция: субагент «Git Workflow» (git-workflow)

Субагент **git-workflow** предназначен для анализа изменений в репозитории, формирования и выполнения коммитов **на русском языке** в соответствии с общепринятыми практиками, а также пуша в удалённый репозиторий.

## Назначение

- Анализировать изменения: `git status`, `git diff`, понимать типы изменений (добавлено, изменено, удалено).
- Предлагать логичную группировку изменений в коммиты (не всё в один коммит при разнородных правках).
- Формировать сообщения коммитов на русском: императив, краткая тема (~50–72 символа), при необходимости тело; опционально префиксы `docs:`, `fix:`, `feat:` и т.д.
- Не коммитить служебное и временное (проверка .gitignore, исключение .env с секретами, лишних бинарников).
- Выполнять пуш в удалённый репозиторий с учётом необходимости pull/rebase при необходимости.

По умолчанию субагент **предлагает план коммитов** и выполняет `git add` / `git commit` / `git push` только после подтверждения пользователя или по явной инструкции («закоммить и запушь»).

## Когда использовать

- Нужно разобрать текущие изменения и разбить их на осмысленные коммиты.
- Нужно сформулировать сообщения коммитов на русском по единому стилю.
- Нужно закоммитить и запушить изменения, в том числе с предварительным pull/rebase.
- Нужно проверить, не попадают ли в коммит лишние или секретные файлы.

## Как вызвать в Cursor

Указать субагента **git-workflow** в чате или при постановке задачи, например:

- «Используй субагент git-workflow: проанализируй изменения и предложи план коммитов».
- «git-workflow: что изменено и как лучше разбить на коммиты?»
- «git-workflow: закоммить и запушь в origin main» — выполнить коммиты и пуш сразу.
- «Субагент git-workflow: предложи сообщения для коммитов на русском и выполни коммиты после моего подтверждения».

Файл субагента: `.cursor/agents/git-workflow.md`.

## Входные данные

- Текущее состояние репозитория (субагент сам выполняет `git status`, `git diff` при необходимости).
- Желание пользователя: только план и сообщения **или** сразу выполнить коммиты и пуш.
- При необходимости — указание ветки и remote (например `origin main`); иначе субагент определяет по контексту.

## Выходы / результат

- **Режим «предложить план»**: краткое резюме изменений (ветка, файлы), нумерованный план коммитов с сообщениями на русском и списками файлов; предупреждения о файлах, которые не стоит коммитить.
- **Режим «выполнить»**: те же коммиты и сообщения плюс выполненные команды `git add`, `git commit`, при необходимости `git pull --rebase`, затем `git push`; краткий отчёт о результате.

## Примеры сообщений коммитов на русском

- `docs: обновить контекстную диаграмму MVP`
- `feat: добавить страницу бронирования с выбором зоны`
- `fix: исправить расчёт стоимости при скидке ОВЗ`
- `refactor: вынести проверку тарифов в отдельный модуль`
- `chore: обновить зависимости в package.json`
- `Добавить валидацию номера договора в форме клиента`
- `Исправить отображение даты окончания бронирования в истории сессий`

Императив: «добавить», «исправить», «обновить» — не «добавлено», «исправлено».

## Ограничения

- Конфликты после pull/rebase субагент не разрешает автоматически — сообщает пользователю.
- Секреты и чувствительные данные не должны попадать в репозиторий; субагент предупреждает о подозрительных путях (например `.env`), но окончательное решение — за пользователем.
