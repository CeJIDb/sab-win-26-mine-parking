# План: унификация формата таблиц маппинга ЮKassa

**Дата**: 2026-04-30
**Задача**: привести таблицы в [docs/architecture/integration/yookassa-data-mapping.md](../docs/architecture/integration/yookassa-data-mapping.md) к каноничному 5-колонному формату с двойной шапкой; устранить дубли между «Табличным маппингом» и «Маппингом запроса/ответа/webhook/возврата»; добавить явные заголовки процессов; отдельным отчетом зафиксировать расхождения с другими артефактами.
**Время**: 3-4 часа

## Зачем именно так

Сейчас маппинг ЮKassa в одном файле описан в двух параллельных нотациях: «Табличный маппинг в формате источник -> приемник» (7 колонок: Объект / Поле / Тип) и серия разделов «Маппинг запроса/ответа/webhook/возврата» (4 колонки: Поле ЮKassa / Откуда / Таблица-поле / Правило). Обе формы покрывают ту же предметку — это дубль и риск рассогласования при правках.

На лекции зафиксирован каноничный для курса формат маппинга полей: **5 колонок, двойная шапка** (`Объект/Атрибут | Тип, значение` со стороны источника и приемника + `Комментарий`). Под него уже подходит подсекция «Наш запрос payment.payments -> ЮKassa create payment», но не до конца — в ней одна шапка, и колонки слегка иначе названы.

**Решения, принятые в диалоге до начала работы:**

- **Один процесс — одна таблица.** Каждый из шести процессов интеграции с ЮKassa получает отдельную таблицу с явным заголовком: создание платежа, ответ ЮKassa по платежу, HTTP-уведомление по платежу (`payment.*`), HTTP-уведомление по возврату (`refund.succeeded`), запрос возврата, ответ по возврату. **Webhook-события платежа и возврата разводятся в разные таблицы**, потому что объекты `Payment` и `Refund` имеют разную структуру и разный набор полей; смешивание возродит тот же дубль, ради устранения которого затеяна работа.
- **Слияние секций.** «Табличный маппинг» как самостоятельный раздел убирается. Его содержимое поглощается соответствующими процессными таблицами.
- **Двойная шапка — фиксированный шаблон.** Markdown не поддерживает colspan. Применяется визуальный прием: первая строка — групповые заголовки `| Источник | | Приемник | | Комментарий |`, вторая строка — обязательный markdown-разделитель `| --- | --- | --- | --- | --- |`, третья строка — подзаголовки колонок `| Объект/атрибут | Тип, значение | Объект/атрибут | Тип, значение | |`, далее идут данные. Третья строка markdown трактует как первую строку данных, не как шапку — это сознательный визуальный компромисс, ради соответствия лекционному формату. **Шаблон оформляется блоком `<!-- prettier-ignore-start --> ... <!-- prettier-ignore-end -->`, чтобы `format-on-write` (Prettier) не нормализовал ширины колонок и не сломал ровнение пустых ячеек.** Все 5 (на самом деле 6) процессных таблиц делаются строго по этому шаблону, без вариаций.
- **Конвенция ячеек:** «Объект/Атрибут» — имя объекта/поля (`payments.amount_minor`, `Payment.amount.value`); «Тип, значение» — тип, а если значение — производное или константа, оно идет сюда (`BIGINT`, `string`, `Оплата бронирования №{bookings.booking_number}`, `0`, `1`). Если объекта на стороне источника нет (литерал/derived) — ячейка «Объект/Атрибут» пуста.
- **Маппинг значений — 4 колонки.** Это маппинг enum-значений, а не полей; смешивание с 5-колонным форматом теряет смысл и плодит пустые ячейки. Формат: «исходное значение / целевое значение / таблица-поле / комментарий». **Колонка «таблица-поле» сохраняется**, потому что без нее теряется контекст — какому полю в БД принадлежит целевое значение (`payment.payments.status` против `payment.refunds.status`). Применяется к маппингу статусов платежа, статусов возврата и `payment_method.type`.
- **Скоуп — только архитектура и трассировка.** [docs/specs/integration/integration-requirements.md](../docs/specs/integration/integration-requirements.md) не правится: INT-004 / INT-006 не фиксируют форму полей, маппинг — деталь архитектуры. Расхождения, обнаруженные при сверке с ЮKassa и с другими артефактами репо, фиксируются отдельным отчетом (Фаза 7), а не правками specs в этом плане.
- **Источник истины по ЮKassa — официальная документация API: <https://yookassa.ru/developers/api>.** Каждое поле, тип, обязательность, формат значения и набор статусов в новых таблицах должны быть сверены с актуальной версией этой документации на дату 2026-04-30. Догадки и устаревшие значения из текущего файла не переносятся «как есть» без проверки. Подразделы документации, которые точно нужно открыть: создание платежа (`POST /v3/payments`), объект `Payment`, входящие уведомления (webhook), создание возврата (`POST /v3/refunds`), объект `Refund`, чек (`receipt`), способы оплаты (`payment_method.type`). При расхождении между текущим [yookassa-data-mapping.md](../docs/architecture/integration/yookassa-data-mapping.md) и документацией ЮKassa приоритет — у документации; расхождение фиксируется в комментарии ячейки и в отчете Фазы 7.
- **Судьба двух «маленьких» таблиц текущего файла.** «Рекомендуемое преобразование суммы» (числовая иллюстрация формата суммы) переносится в раздел «Правила валидации и хранения» как пример к правилу про minor/major units, отдельной таблицей не остается. «События ЮKassa и реакция платформы» (бизнес-реакция платформы на события) сохраняется как самостоятельный раздел в текущем 2-колонном виде — это не маппинг полей и не маппинг значений, а описание реакции платформы; в новые форматы оно не вписывается и не должно.

Отказались от:

- Сохранения раздела «Табличный маппинг» как сводного представления — дубль с процессными таблицами после унификации становится бессмысленным.
- 7-колонного формата (Объект / Поле / Тип) — лекция фиксирует 5-колонный, и это зафиксировано как канон курса.
- Единой шапки без группы — не соответствует образцу лекции.
- Включения статусов в общий 5-колонный формат — теряется смысл колонки «Тип» для enum-значений.
- Объединения webhook'ов платежа и возврата в одну таблицу — нарушает принцип «один процесс — одна таблица» и возвращает дубль с таблицами «ответ по платежу» и «ответ по возврату».
- 3-колонного формата маппинга значений (без «таблица-поле») — терялась бы привязка статуса к конкретному полю БД.

## Шаблон двойной шапки (5-колонный процессный маппинг)

Все шесть процессных таблиц делаются строго по этому шаблону. Блок `prettier-ignore` обязателен — без него Prettier на `format-on-write` нормализует ширины и сломает ровнение пустых ячеек.

```markdown
<!-- prettier-ignore-start -->

| Источник         |                            | Приемник         |                            | Комментарий |
| ---------------- | -------------------------- | ---------------- | -------------------------- | ----------- |
| Объект/атрибут   | Тип, значение              | Объект/атрибут   | Тип, значение              |             |
| `payments.amount_minor` | `BIGINT`            | `Payment.amount.value` | `string`             | Конвертируем из minor в `"123.45"` |

<!-- prettier-ignore-end -->
```

Замечания:

- Строка 1 — групповые заголовки. Источник занимает колонки 1-2, приемник — колонки 3-4, комментарий — колонка 5. Колонки 2 и 4 в этой строке остаются пустыми (визуальная имитация colspan).
- Строка 2 — обязательный markdown-разделитель ровно из 5 ячеек `---`.
- Строка 3 — подзаголовки колонок. Markdown трактует ее как первую строку данных, но визуально она читается как нижняя часть «двойной шапки». Это сознательный компромисс.
- Строки 4+ — данные.
- Если объект источника отсутствует (литерал, derived, конфигурация) — первая ячейка пуста, во второй идет константа или формула.

## Цель

- В [docs/architecture/integration/yookassa-data-mapping.md](../docs/architecture/integration/yookassa-data-mapping.md) шесть процессных маппингов представлены **5-колонными таблицами с двойной шапкой по единому шаблону** (см. выше):
  1. Создание платежа (`POST /v3/payments`) — направление: модуль Оплата → ЮKassa.
  2. Ответ ЮKassa по платежу — направление: ЮKassa → модуль Оплата.
  3. HTTP-уведомление по платежу (`payment.waiting_for_capture`, `payment.succeeded`, `payment.canceled`) — направление: ЮKassa → модуль Оплата.
  4. HTTP-уведомление по возврату (`refund.succeeded`) — направление: ЮKassa → модуль Оплата.
  5. Запрос возврата (`POST /v3/refunds`) — направление: модуль Оплата → ЮKassa.
  6. Ответ ЮKassa по возврату — направление: ЮKassa → модуль Оплата.
- У каждой таблицы — явный H3-заголовок с названием процесса и направления потока.
- Раздел «Табличный маппинг в формате источник -> приемник» удален, его содержимое перенесено в процессные таблицы.
- Разделы «Маппинг статусов» и «Маппинг `payment_method.type`» сохранены как «маппинг значений» с **единым 4-колонным форматом** (исходное значение / целевое значение / таблица-поле / комментарий).
- «Рекомендуемое преобразование суммы» — переехало в «Правила валидации и хранения» как иллюстрация.
- «События ЮKassa и реакция платформы» — сохранен как самостоятельный 2-колонный раздел про реакцию платформы (это не маппинг).
- Покрытие полей не меньше текущего — все поля из текущих 4-колонных таблиц и 7-колонных «Табличных» сохранены или явно сняты с обоснованием.
- Все внутренние якоря (оглавление файла) и обратные ссылки из других документов репо обновлены под новую структуру H2/H3.
- Сформирован отдельный отчет [docs/architecture/integration/yookassa-mapping-divergences.md](../docs/architecture/integration/yookassa-mapping-divergences.md) о расхождениях текущего маппинга с другими артефактами (specs, ERD, sequence-диаграмма, UC-10.2, payload/schema) и с актуальной документацией ЮKassa.
- Обновлен [docs/process/traceability-matrix-log.md](../docs/process/traceability-matrix-log.md): запись формата «Источник → Требование → Изменения → Проверка → Доказательство» с ссылкой на INT-004 / INT-006.
- `npm run lint:md`, `npm run lint:md-links`, `npm run ci:check` зеленые.
- Имя файла плана и ретро — латиница kebab-case, без буквы «е» в позиции «ё».
- В конце сессии — ретро в `docs/process/retro/2026-04-30-yookassa-mapping-table-format-alignment.md`.

## Открытые риски

- **Prettier на `format-on-write` ломает ровнение пустых ячеек двойной шапки.** Митигация: каждую процессную таблицу обернуть в `<!-- prettier-ignore-start --> ... <!-- prettier-ignore-end -->`. На Фазе 8 проверить, что после прогона `npm run format:check` или ручного `npm run format` блоки не съехали.
- **Двойная шапка может ломать `markdownlint` правило `MD056` (table-column-count).** В текущем `.markdownlint.jsonc` `MD056` отключен. Сверить, что отключение сохраняется, и не полагаться на него для других таблиц.
- **Ширина строк в больших таблицах.** `MD013` глобально отключен в `.markdownlint.jsonc` — риска нет, но проверить.
- **Сверка с ЮKassa может найти противоречия с `docs/specs/`.** План запрещает правки specs; все противоречия идут в отчет Фазы 7 как материал для отдельного плана. Это сознательный выбор, чтобы не разрастать скоуп.
- **Битые якоря после переименования заголовков.** `MD051` отключен в `.markdownlint.jsonc`, `lint:md-links` не валидирует внутрифайловые `#anchor`. Митигация: ручной `grep -rn` по всему репо и сверка глазами на Фазе 6.
- **При сворачивании дублей возможны конфликты значений между «Табличным маппингом» и «Маппингом запроса/ответа»** (например, `BIGINT` vs `decimal string` для одного логического поля) — на Фазе 0 фиксируем все такие точки и согласуем версию-источник; конфликты идут в отчет Фазы 7.
