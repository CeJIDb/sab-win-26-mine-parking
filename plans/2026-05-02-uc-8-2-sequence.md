# Подплан Фазы 4: UC-8.2 — UML Sequence (Mermaid + PlantUML)

**Дата**: 2026-05-02
**Родительский план**: [2026-05-02-uc-8-2-edo-integration.md](2026-05-02-uc-8-2-edo-integration.md)
**Задача**: создать [docs/architecture/integration/sequence-uc-8-2-create-contract.md](../docs/architecture/integration/sequence-uc-8-2-create-contract.md) — UML Sequence для сценария UC-8.2 «Создать договор с ЮЛ через ЭДО» в двух нотациях: inline Mermaid (как [UC-12.1](../docs/architecture/integration/sequence-uc-12-1-pass-auto-identification-entry.md)) и PlantUML (`.puml`-исходник + PNG-экспорт). Диаграмма покрывает основной поток (~18 шагов) и 6 расширений (`3а`, `7а`, `10а`, `10б`, `12а`, `12б`).
**Время**: ~1.5–2 ч (с учетом двух нотаций и PNG-экспорта)

## Зачем именно так

Sequence — четвертый deliverable из 9 в наборе UC-8.2. Он опирается на уже зафиксированные в Фазах 1–3 артефакты: бизнес-сценарий [UC-8.2](../docs/artifacts/use-case/uc-8-2-create-contract-legal-entity.md), требования [INT-014 / INT-015](../docs/specs/integration/integration-requirements.md), [Блок 3 регламента](../docs/architecture/integration/is-interaction-regulation.md). Дает картину «как именно сообщения движутся между PWA, Сервисом Договоров, БД, Notification Service / Adapter, ЭДО и Сервисом уведомлений во времени», которая в UC и регламенте отсутствует.

Стиль и оформление воспроизводят [UC-12.1 Sequence](../docs/architecture/integration/sequence-uc-12-1-pass-auto-identification-entry.md) — единственный Mermaid-образец в проекте (другой Sequence — UC-10.2 — это draw.io PNG, не Mermaid). Outbox-механика для уведомлений показывается явно, как в UC-12.1, через `INSERT outbox_events (...)` + `CDC / Outbox Polling`. Это согласуется с [ADR-003](../docs/architecture/adr/adr-003-modular-monolith.md) инв. 4 (transactional outbox для развязки модулей).

PlantUML добавляется как вторая нотация на тот же сценарий — учебная цель курса (показать обе нотации UML Sequence на одном артефакте). PlantUML в проекте раньше не использовался, поэтому подплан фиксирует и инфраструктурную часть: расположение `.puml`-исходника, PNG-экспорт, ручной рендер через [plantuml.com/plantuml](https://plantuml.com/plantuml).

### Состав основного потока (по UC-8.2 шагам 1–18)

Mermaid и PlantUML диаграммы воспроизводят последовательность шагов UC-8.2 буквально (не по «модельной семантике», а по нумерации UC):

- **Шаги 1–3** UC: PWA → API Gateway → `cs` (`POST /api/contracts` с параметрами формы); валидация / расчет стоимости — внутри `cs` (без отдельных стрелок к Pricing — это другой уровень детализации).
- **Шаг 5** (формирование документа): `cs->>cps: GET /clients/{id}/organization (через view)` для подстановки реквизитов ЮЛ → `cps-->>cs: реквизиты`.
- **Шаг 4** (создание `[договор]` в `статус=черновик`): `cs->>db: INSERT contracts (status=DRAFT, ...)`.
- **Шаги 6–7**: `cs->>edo: POST /documents (типДокумента, маршрут=Клиент→Владелец, внутреннийИдДоговора, реквизиты)` → `edo-->>cs: 200 OK (идДокументаЭДО, статусПриема)`. С `Note over cs,edo` про идемпотентность по `внутреннийИдДоговора`.
- **Шаги 8–9** (статус=на подписании клиентом + уведомление): `cs->>db: UPDATE contracts SET status=AWAITING_CLIENT_SIGNATURE` + `INSERT outbox_events (type=ДоговорОтправленНаПодпись, получатель=клиент)` (одной транзакцией) → асинхронно `db-->>ns: CDC / Outbox Polling` → `ns->>na: ...` → `na->>nsvc: send`.
- **Ожидание подписи клиента**: `Note over cs,edo` + `...`. Сразу за ним — `opt polling fallback (если webhook не доставлен в течение настраиваемого интервала)`: `cs->>edo: GET /documents/{id}/status` → `edo-->>cs: статус`.
- **Шаги 10–11**: основной случай — `edo->>cs: webhook: document.signed.by.client (идДокументаЭДО, моментПодписи, идентификаторПодписанта)` → `cs->>db: UPDATE contracts SET status=AWAITING_OWNER_SIGNATURE, client_signed_at=...`.
- **Ожидание финальной подписи**: Note + `...` + `opt polling fallback`.
- **Шаги 12–14**: `edo->>cs: webhook: document.fully.signed (идДокументаЭДО, содержимое, моментФинальнойПодписи)` → `cs->>db: file storage save (ссылка)` → `cs->>db: UPDATE contracts SET status=ACTIVE, document_file_ref=..., owner_signed_at=...`.
- **Шаг 15** (бронирования): `cs->>db: INSERT bookings (...)` для всех ПМ из формы (без раскрытия Booking Service на этом уровне; формулировка в Note).
- **Шаги 16–17** (два уведомления): `cs->>db: INSERT outbox_events (type=ДоговорЗаключен, получатель=клиент)` + `INSERT outbox_events (type=ДоговорЗаключен, получатель=Управляющий)` → асинхронно две доставки через ns / na / nsvc.

## Содержательные правила (общие для обеих нотаций)

- **Имена participants** — синхронизированы с C4 L3 / DFD L1, без P-номеров и эмодзи. `Сервис Договоров` (а не «Договор» из INT-\* или «Contract Service» из C4).
- **Стрелки** — гибрид: латинские эндпойнты (`POST /api/contracts`, `GET /clients/{id}/organization`, `GET /documents/{id}/status`, `webhook: document.signed.by.client`, `INSERT outbox_events (...)`, `UPDATE contracts SET ...`); русский текст для семантических пометок в `Note over`.
- **Стиль блоков** — по [UC-12.1](../docs/architecture/integration/sequence-uc-12-1-pass-auto-identification-entry.md): `rect rgb(255, 235, 235)` + `break` для terminal failure (расширения 7а, 10а, 10б, 12а, 12б), `alt` для разветвлений с не-терминальным else (3а), `opt` для опциональных веток (polling fallback).
- **Outbox-механика** — `cs->>db: INSERT outbox_events (aggregate=CONTRACT, type=..., получатель=...)`. Асинхронная доставка показана пунктирной стрелкой `db-->>ns: CDC / Outbox Polling (асинхронно)`, далее `ns->>na: ...` и `na->>nsvc: send (...)`. Транзакционная семантика — `Note over cs,db: Транзакционный блок` (UPDATE + INSERT outbox в одной транзакции).
- **SLA 7 календарных дней** — фиксируется в `Note over cs,edo: Ожидание подписи клиента (до 7 календарных дней — INT-014 SLA)`. Не дублируем 7-дневный лимит в стрелках, не зашиваем в `idle`-таймер.
- **Идемпотентность повторов в ЭДО** — `Note over cs,edo: Повторы отправки идемпотентны по внутреннийИдДоговора (INT-014)` рядом с шагом 6.
- **Уведомления INT-015** — в стрелке только тип события + получатель: `→ ns: ДоговорЗаключен (получатель=клиент)`, `→ ns: ДоговорЗаключен (получатель=Управляющий)`, `→ ns: ДоговорОтклонен (получатель=клиент)`, `→ ns: ДоговорПросрочен (получатель=клиент)`. Полный набор атрибутов — в [INT-015](../docs/specs/integration/integration-requirements.md).
