# План: ДЗ по интеграционному UC-8.2 «Создать договор с ЮЛ через ЭДО»

**Дата**: 2026-05-02
**Задача**: подготовить полный комплект учебных артефактов по интеграционному use case UC-8.2 (создание договора с юридическим лицом с подписанием через ЭДО) — 9 deliverables по списку курса.
**Время**: ~10–12 ч (распределено на несколько сессий)

## Зачем именно так

Курс требует выбрать один интеграционный UC и проработать его на всю глубину: от пользовательского описания до OpenAPI-спеки и XML-схемы. UC-8.2 выбран после анализа реестра UC, контекстной диаграммы и [DFD L1](../docs/artifacts/dfd-l1.md):

- внешние системы: **ЭДО** (E12 — нативный XML-канал, СБИС/Диадок) и **Сервис уведомлений** (E8 — JSON SMS/email);
- внутренние модули: **`Сервис Договоров` (P14)** владеет XML-сериализацией для E12 ЭДО самостоятельно (отдельный адаптер не вводится — поток обмена соответствует [DFD L1](../docs/artifacts/dfd-l1.md) F03/F04) и публикует события для `Notification Service` через transactional outbox ([ADR-003](../docs/architecture/adr/adr-003-modular-monolith.md) инв. 4);
- асинхрон с ожиданием подписи контрагента (богатый материал для UML Sequence);
- XML+XSD ложатся на формат документов ЭДО органично, не «по желанию».

DADATA-интеграция (автозаполнение реквизитов по ИНН, [ADR-004](../docs/architecture/adr/adr-004-dadata-organization-lookup.md)) **исключена из scope ДЗ** по решению автора — оставляем только ЭДО + Notification как 2 внешние системы.

5. JSON-пример + JSON Schema — `docs/architecture/integration/payload-uc-8-2-create-contract.md` и `schema-uc-8-2-create-contract.md` для внутреннего REST `POST /api/contracts`.

## Scope

**Входит:**

- Интеграционный UC-8.2 во всём комплекте 9 deliverables.
- 2 внешних: ЭДО (E12), Сервис уведомлений (E8).
- Внутренний поток `Сервис Договоров → Notification Service` через transactional outbox — раскрывается на уровне UML Sequence (Фаза 4); на уровне ФТ как отдельное `INT-*` **не выносится** (после удаления INT-016/017 решением 2026-05-02 — см. CHG-20260502-005).
- Связь `Сервис Договоров ↔ Client Profile` (read через view) — упоминается в UC и Sequence как контекстная, но в регламент/ФТ как «третья интеграция» **не выносится**.

**Не входит:**

- Правка [dfd-l1.md](../docs/artifacts/dfd-l1.md) и [c4-diagrams.md](../docs/architecture/c4/c4-diagrams.md) — принятые артефакты, не трогаем.
- DADATA-интеграция — исключена.
- UC-8.5 (расторжение договора), UC-8.4 (изменение договора) — отдельные UC, не входят в это ДЗ.
- Реальная конфигурация СБИС/Диадок-провайдера — XSD проектируется как учебный, не привязан к конкретному вендору.

**Ключевые решения, принятые в Фазе 1** (важны для последующих фаз):

- **Граничное условие**: создание бронирований (шаг 15 UC) — часть UC-8.2, не передача в UC-7.1. Поток `Contract → Facility` упоминается только бизнес-формулировкой «создает бронирования» без раскрытия модулей. В регламенте/ФТ как «третья внутренняя интеграция» **не выносится** — единственная внутренняя связь, которая раскрывается на уровне UML Sequence (Фаза 4) — `Сервис Договоров → Notification Service` через outbox.
- **Уведомление Владельца** на стороне платформы отсутствует — это делает ЭДО (его кабинет уведомляет автоматически). Промежуточно подписанный документ к нам не приезжает.
- **SLA подписи** — 7 календарных дней на каждый этап (клиент → Владелец). Заказчиком не озвучивалось, выбрано на основе практики российских ЭДО (Диадок/СБИС: 5–10 дней). Параметр выносится в конфигурацию.
- **Запись `CHG-*` в журнал трассировки** делается **в Фазе 10**, одной общей записью на весь набор 9 артефактов + правки концептуальной модели и ERD (а не пофазно).

### Фаза 2 завершена 2026-05-02

Подплан: [2026-05-02-uc-8-2-functional-requirements.md](2026-05-02-uc-8-2-functional-requirements.md).

**Артефакты:**

- [docs/specs/integration/integration-requirements.md](../docs/specs/integration/integration-requirements.md) — расширен на 4 новых `INT-*` (на момент завершения Фазы 2; постфактум 2 из них удалены — см. постскриптум):
  - `INT-014` (6 подзаписей) — Договор ↔ ЭДО (внешняя): исходящий документ с маршрутом, подтверждение приема, события подписи клиента, финальный документ, события отказа и просрочки. SLA 7 дней. Webhook + polling fallback.
  - `INT-015` (4 подзаписи) — Договор → АгентУведомлений (внешняя): события отправки на подпись, заключения (клиент + Управляющий), отклонения, просрочки. Владелец платформой не уведомляется.
  - `INT-016` (3 подзаписи) — Договор ↔ ЭДО Adapter (внутренняя, учебный артефакт): команда подписания, трансляция событий статуса, передача финального документа. Пометка про локальный адаптер.
  - `INT-017` (2 подзаписи) — Договор → Notification outbox (внутренняя): публикация в outbox в одной транзакции, подписка с at-least-once. Transactional outbox по ADR-003 инв. 4.

**Постскриптум 2026-05-02** (после завершения Фазы 3, перед стартом Фазы 4): `INT-016` и `INT-017` удалены из [integration-requirements.md](../docs/specs/integration/integration-requirements.md) вместе с outbox-секцией и пояснительными примечаниями (`CHG-20260502-005`); одновременно отказались от учебного допущения «локальный ЭДО Adapter» — поток обмена с E12 в UML Sequence идет напрямую от `Сервис Договоров` (P14), как в DFD L1 F03/F04. Внутренняя связь `Сервис Договоров → Notification Service` через transactional outbox сохраняется на уровне архитектурных артефактов (Sequence, JSON), но в файле требований больше не закреплена. Мотивация: курсовое требование «2+2 интеграции» жестче, чем требование сохранять учебные допущения; выбрано состояние «2 внешних INT-\* + outbox-механика как архитектурная деталь Sequence».

**Решения, принятые в Фазе 2:**

- Параллельное выполнение 4 агентов (по одному на INT-ID) с последующей сборкой оркестратором — снижает риск гонок за файл.
- Все 4 требования вставлены в один файл `integration-requirements.md` подряд после INT-013, перед `---`.
- Индекс `docs/specs/integration/readme.md` не требует обновки (перечень INT-\* там отсутствует).
- Все линтеры прошли успешно, текст без буквы «ё».

**Ключевые решения, принятые в Фазе 3** (важны для последующих фаз):

- **Терминология «Сервис Договоров» вместо «Договор»** — в колонках источника / приёмника таблицы регламента используются имена контейнеров C4 / DFD L1. Будущие архитектурные артефакты UC-8.2 (Sequence Фазы 4, OpenAPI Фазы 8, регламент Фазы 9) — пишут «Сервис Договоров». В файле требований [integration-requirements.md](../docs/specs/integration/integration-requirements.md) сохраняется логическое имя «Договор» (формат `INT-*`), это допустимое расхождение между уровнями спецификаций и архитектуры.
- **Внутренние модульные связи в регламент не выносятся** — на уровне регламента отображаются только направления обмена с **внешними** системами (ЭДО и Сервис уведомлений). Внутренняя связь `Сервис Договоров → Notification Service` (через outbox) раскрывается на уровне UML Sequence (Фаза 4).
- **SLA 7 календарных дней** — зафиксирован в колонке «Периодичность обмена / событие» строк просрочки (строки 6 и 10), а не в «Передаваемых данных». Согласовано с подходом Блока 2 (триггеры событий — в колонке периодичности).
- **Триггеры строк уведомлений** (строки 7–10) — без ссылок на конкретные UC-шаги / расширения; формулировки бизнес-нейтральные («после успешной передачи документа в ЭДО», «отклонение договора клиентом или Владельцем»). Облегчает поддержку при возможных правках UC.
- **«Изменения (статус документа)»** — единая формулировка для строк отказа и просрочки от ЭДО (по образцу «Изменения (статус операции)» из Блока 2).
