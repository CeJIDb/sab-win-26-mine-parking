# Подплан Фазы 2: UC-8.2 — пользовательские требования к интеграции (ФТ)

**Дата**: 2026-05-02
**Родительский план**: [2026-05-02-uc-8-2-edo-integration.md](2026-05-02-uc-8-2-edo-integration.md)
**Задача**: расширить [docs/specs/integration/integration-requirements.md](../docs/specs/integration/integration-requirements.md) требованиями `INT-*` под UC-8.2 — 2 внешних (ЭДО, Notification) + 2 внутренних (Contract ↔ ЭДО Adapter, Contract → Notification outbox), всего 4 новых ID с атомарными подзаписями по типам взаимодействия.
**Время**: ~1 ч

## Зачем именно так

Курс ДЗ требует 2 внешних + 2 внутренних интеграции — отступать от этой схемы нельзя. В файле уже есть `INT-010/011` (Notification, общее) и `INT-012` (ЭДО, общий контракт по любым документам ЮЛ). UC-8.2 дает более узкий и детализированный контракт: события подписи, маршрут согласования в ЭДО, SLA 7 дней, идемпотентность повторов, локальный ЭДО Adapter (артефакт ДЗ).

Дублирование с существующими `INT-010/011/012` допускается — UC-8.2-специфичные `INT-014/015` дополняют их подсобытиями и параметрами, не отменяют. Внутренние интеграции (`INT-016/017`) — новые, в файле прецедента под локальный ЭДО Adapter нет.

Образец формата атомарных подзаписей — `INT-004` (платежи): один ID + несколько строк под каждую команду/событие. Этот стиль и воспроизводим.

## Структура: ID и подзаписи

### `INT-014` Договор ↔ ЭДО (внешняя, UC-8.2-специфика)

Дополняет `INT-012` подсобытиями маршрута согласования. Подзаписи (черновой состав, финал — в файле):

- `Договор.ИсходящийДокументСМаршрутом.ЭДО` — передача документа с маршрутом «клиент → Владелец», пакет атрибутов: типДокумента, содержимое/ссылка, внутреннийИдДоговора, реквизитыКонтрагента, маршрутПодписания.
- `ЭДО.ПодтверждениеПриема.Договор` — прием от ЭДО идентификатора документа в ЭДО (синхронный ответ на отправку).
- `ЭДО.СобытиеПодписиКлиента.Договор` — асинхронное событие «документ подписан клиентом» (без полезного документа); атрибуты: идДокументаЭДО, моментПодписи.
- `ЭДО.ФинальныйДокумент.Договор` — асинхронное получение документа с двумя ЭП; атрибуты: идДокументаЭДО, содержимое/ссылка, моментФинальнойПодписи.
- `ЭДО.СобытиеОтказа.Договор` — асинхронное событие отказа (клиент или Владелец отклонил); атрибуты: идДокументаЭДО, сторонаОтказа, моментОтказа.
- `ЭДО.СобытиеПросрочки.Договор` — событие истечения срока подписи (7 календарных дней на каждый этап).

Транспорт асинхронных событий — webhook + polling fallback (полл по идДокументаЭДО, если webhook не доставлен в течение настраиваемого интервала). SLA подписи — **7 календарных дней** на каждый этап (клиент, Владелец).

### `INT-015` АгентУведомлений ↔ СервисУведомлений (внешняя, UC-8.2-специфика)

Дополняет `INT-010/011` UC-8.2-специфичными значениями `типСобытияИсточник`. Подзаписи:

- `Договор.СобытиеОтправкиНаПодпись.АгентУведомлений` — клиенту: «договор отправлен на подпись в [ЭДО]» (после успешной передачи в ЭДО, шаг 9 UC).
- `Договор.СобытиеЗаключения.АгентУведомлений` — клиенту + Управляющему: «договор заключен» (после получения финального документа, шаги 16–17 UC).
- `Договор.СобытиеОтклонения.АгентУведомлений` — клиенту: «договор отклонен» (после `ЭДО.СобытиеОтказа.Договор`).
- `Договор.СобытиеПросрочки.АгентУведомлений` — клиенту: «договор просрочен» (после `ЭДО.СобытиеПросрочки.Договор`).

Владелец **не уведомляется платформой** — это делает ЭДО (его кабинет уведомляет автоматически). Зафиксировать это в формулировке требования.

### `INT-016` Договор ↔ ЭДО Adapter (внутренняя, локальный адаптер ДЗ)

Локальный ЭДО Adapter — артефакт ДЗ UC-8.2 (см. шапку [мастер-плана](2026-05-02-uc-8-2-edo-integration.md)). В файле требований это отражается явной пометкой «локальный адаптер в рамках ДЗ UC-8.2». Подзаписи:

- `Договор.КомандаПодписания.ЭДОАдаптер` — Договор передает адаптеру документ + маршрут согласования; адаптер отвечает за XML-сериализацию и протокол E12.
- `ЭДОАдаптер.СобытиеСтатуса.Договор` — адаптер транслирует события ЭДО (подпись клиента, отказ, просрочка) в нормализованный формат для модуля Договор.
- `ЭДОАдаптер.ФинальныйДокумент.Договор` — адаптер передает финальный документ с двумя ЭП (после `document.fully.signed` от ЭДО).

### `INT-017` Договор → АгентУведомлений (внутренняя, через outbox)

По [ADR-003](../docs/architecture/adr/adr-003-modular-monolith.md) инв. 4 — transactional outbox для развязки модулей. Подзаписи:

- `Договор.ПубликацияСобытия.OutboxУведомлений` — Договор публикует событие в outbox в одной транзакции с изменением `[договор].статус`. Атрибуты события: типСобытия (отправлен/заключен/отклонен/просрочен), идДоговора, идКлиента, идУправляющего (если применимо), параметрыШаблона.
- `АгентУведомлений.ПодпискаНаСобытия.OutboxУведомлений` — АгентУведомлений потребляет события с семантикой at-least-once и ключИдемпотентности (защита от повторов на уровне внешней доставки — см. `INT-011`).

## Содержательные правила (общие для всех 4 ID)

- **Стиль формулировок** — по образцу существующего файла: «Модуль X должен передавать/получать в/от Y …» + bullet-список атрибутов.
- **Имена атрибутов** — в формате camelCase на русском (как в `INT-001/003/004`), без латинских смесей и без `idВнешнегоЭДО`-стиля.
- **Идемпотентность повторов** — упоминается в `INT-014` (для отправки документа: повторная отправка с тем же `внутреннийИдДоговора` не должна порождать дубликатов в ЭДО) и в `INT-017` (для outbox: ключИдемпотентности на уровне публикации события).
- **SLA 7 календарных дней** — упоминается в `INT-014` явно (`ЭДО.СобытиеПросрочки.Договор`). В UC-8.2 этот параметр был помечен как «выносится в конфигурацию», но в требовании фиксируем число — это решение Q5.
- **Маркировка локального ЭДО Adapter** — в шапке `INT-016` явная пометка: «Локальный ЭДО Adapter — учебный артефакт UC-8.2; в архитектуру проекта не входит, см. [мастер-план UC-8.2](../../../plans/2026-05-02-uc-8-2-edo-integration.md)».

## Scope

**Входит:**

- 4 новых `INT-*` (014/015/016/017) с атомарными подзаписями.
- Маркировка локального ЭДО Adapter в `INT-016`.
- Обновление индекса [docs/specs/integration/readme.md](../docs/specs/integration/readme.md), если он перечисляет `INT-*` (проверить).
- Прогон линтеров: `npm run lint:md`, `npm run lint:md-links`, `npm run lint:file-names`.

**Не входит:**

- Правки `INT-010/011/012` — оставляем как есть (UC-8.2 дополняет их новыми ID, не переписывает).
- Маппинги данных в формате `yookassa-data-mapping.md` — для UC-8.2 такой файл будет в Фазе 5/6 (JSON/XML), здесь — только текстовые требования.
- Запись `CHG-*` в [traceability-matrix-log.md](../docs/process/traceability-matrix-log.md) — делает мастер-план в Фазе 10 (одной общей записью на весь набор 9 артефактов).
- Правки UC-8.2 — он уже зафиксирован Фазой 1.
- Sequence / JSON / XML / Postman / Swagger — Фазы 4–8.
- Регламент v1/v2 — Фазы 3/9 (там и появятся ссылки на `INT-014/015/016/017`).
- **Синхронизация buildin.ai** — пользователь правит страницу «Требования к интеграции» вручную после Фазы 2 (вне scope подплана и репо).
