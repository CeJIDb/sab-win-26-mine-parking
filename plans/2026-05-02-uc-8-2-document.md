## Содержательный каркас (что именно пишем)

### Действующие лица

- **Клиент ЮЛ** — инициатор, у которого уже есть профиль в системе, КЭП в кабинете ЭДО.
- **Система (платформа)** — оркестратор: генерирует договор, отправляет в ЭДО с маршрутом подписания, реагирует на события статусов от ЭДО, забирает финальный документ, создает бронирования.
- **Владелец** — активный актор со стороны платформы: подписывает договор своей КЭП в кабинете того же внешнего ЭДО (E12), что и клиент (или через роуминг). **Уведомление о необходимости подписать ему отправляет ЭДО, не наша система** — Владелец работает в кабинете ЭДО. TO-BE-решение, см. [interview-3](../docs/interviews/protocols/interview-protocol-3-2026-02-04-v01.md).
- **ЭДО (E12)** — внешний оператор (СБИС/Диадок) для подписания и обмена документами; здесь же оркестрирует **маршрут согласования** клиент→Владелец.
- **Сервис уведомлений (E8)** — внешний канал для информирования клиента и Управляющего (Владельца не уведомляем — это делает ЭДО).
- **Управляющий** — постусловный наблюдатель (получает уведомление о новом подписанном договоре). В подписании не участвует.

### Предусловия (черновой список, дошлифуется при написании)

- Клиент ЮЛ авторизован в ЛК.
- В профиле клиента заполнены **данные организации** (UC-2.13 «Добавить данные организации»).
- В профиле клиента заполнены **банковские реквизиты организации** (хотя бы один счет, отмеченный как основной).
- В профиле клиента есть хотя бы одно **ТС** (UC-3.1 «Добавить ТС»).
- Клиент подключен к одному из операторов ЭДО, поддерживаемых платформой (или есть роуминг с тем оператором, где работает Владелец).
- У Владельца есть аккаунт и действующая КЭП в том же операторе ЭДО (или совместимом через роуминг) для подписания договоров со стороны платформы.
- Существует опубликованный шаблон договора долгосрочной аренды.
- Доступны для бронирования машиноместа в нужном секторе на нужный срок.

Конкретный состав полей «данных организации» и «банковских реквизитов» в UC **не дублируем** — отсылаемся к концептуальной модели ([Организация](../docs/artifacts/conceptual-model-with-attributes.md), сущность «Организация» и «Банковские реквизиты организации»). В предусловии ограничиваемся формулировкой «заполнены данные организации / банковские реквизиты». Атрибуты сверяли с ERD ([organizations / organization_bank_accounts](../docs/architecture/database/erd/erd-normalized-er-model.md)) — расхождений нет.

КЭП организации в предусловиях **не фигурирует** — подписание делает Владелец своей КЭП в кабинете ЭДО (Сценарий Б, см. решение №3).

### Триггер

Клиент ЮЛ в ЛК переходит в раздел «Создать договор».

Заполнение формы (срок, тариф, ТС, машиноместа) и подтверждение «Заключить договор» — это **шаги основного потока**, а не часть триггера. Триггер фиксирует только момент инициации сценария.

Поля формы — по [contracts-create.njk](../ui/templates/pages/contracts-create.njk):

- **Срок действия договора**: дата начала, дата окончания.
- **Тариф по договору**: выбор из доступных опубликованных тарифов долгосрочной аренды.
- **ТС по договору**: мультивыбор из ТС в профиле клиента (`vehicles.client_id`).
- **Машиноместа по договору**: мультивыбор из доступных к бронированию ПМ.

Реквизиты ЮЛ в форме **не запрашиваются** — берутся из профиля (см. предусловия).

### Постусловие (целевое финальное состояние)

- `[договор].статус` = `активен`.
- `[договор].файлДоговора` указывает на сохраненный финальный документ с двумя ЭП.
- `[договор].датаПодписанияКлиентом` и `[договор].датаПодписанияПарковкой` заполнены.
- Клиент получил уведомление об успешном заключении договора и доступности в ЛК.
- Управляющий получил уведомление о новом договоре.
- На указанные клиентом машиноместа созданы бронирования на весь срок договора.

### Основной поток (черновая структура — финал в самом файле)

**Принципы**:

- Описываем только то, что наблюдает система. Действия в кабинете ЭДО (Клиент открывает документ и подписывает; Владелец открывает документ и подписывает) системе не видны — шагами не описываются.
- Маршрут согласования живет в ЭДО. Платформа отправляет документ **один раз** с маршрутом «клиент → Владелец», далее реагирует на события статусов и забирает финальный документ. Промежуточно подписанный документ к нам не приезжает.
- Уведомление Владельца о необходимости подписать — за ЭДО (его кабинет уведомляет автоматически). Наша система Владельцу ничего не шлет.

Уровень абстракции — UC: единственный системный актор «Система». Внутренние модули (Pricing / Facility / Contract Service / ЭДО Adapter) в шагах не упоминаем — это уровень Sequence (Фаза 4). Формат документа (XML) тоже опускаем — это уровень XSD/Sequence. **Принцип: один шаг — одно действие.** Атрибуты `[договор]` — по таблице выше.

1. Клиент в форме «Создать договор» указывает параметры аренды: срок (`датаНачала`, `датаОкончания`), тариф из доступных, ТС из профиля, машиноместа.
2. Система рассчитывает стоимость по выбранному клиентом тарифу и отображает условия (сумма, период, состав ПМ, ТС).
3. Клиент подтверждает «Заключить договор» → [Ok].
4. Система создает `[договор]` со следующими атрибутами:
   - `идентификатор` — автоген;
   - `клиентИд` — ссылка на клиента из сессии;
   - `шаблонДоговораИд` — ссылка на опубликованный шаблон долгосрочной аренды для ЮЛ;
   - `номерДоговора` — автоген по правилу нумерации;
   - `датаНачала`, `датаОкончания` — из формы;
   - `статус` = `черновик`;
   - `файлДоговора` = `null`.
5. Система формирует документ договора по `шаблонуДоговора` с подстановкой реквизитов ЮЛ из профиля клиента и параметров из формы.
6. Система передает документ в ЭДО **с маршрутом согласования: клиент → Владелец**.
7. Система получает от ЭДО подтверждение приема документа с идентификатором документа в ЭДО.
8. Система обновляет `[договор]`: `статус` = `на подписании клиентом`.
9. Система отправляет клиенту уведомление «договор отправлен на подпись в [ЭДО]».
10. Система получает от ЭДО событие «документ подписан клиентом» (промежуточно подписанный документ к нам не приезжает).
11. Система обновляет `[договор]`: `статус` = `на подписании парковкой`, `датаПодписанияКлиентом` = момент события из ЭДО.
12. Система получает от ЭДО документ, подписанный обеими сторонами (с двумя ЭП).
13. Система сохраняет финальный документ во внешнем хранилище.
14. Система обновляет `[договор]`: `статус` = `активен`, `файлДоговора` = ссылка на сохраненный документ, `датаПодписанияПарковкой` = момент получения финального документа.
15. Система создает бронирования на указанные клиентом машиноместа на весь срок договора.
16. Система отправляет клиенту уведомление об успешном заключении договора.
17. Система отправляет управляющему уведомление о новом договоре.
18. Сценарий завершается.

### Расширение (исключения по решению из п.5)

Формат — `Nа.` (заголовок жирным) / `Nа1.`, `Nа2.` (подшаги) по образцу [uc-12-1](../docs/artifacts/use-case/uc-12-1-pass-auto-identification-entry.md). Нумерация привязана к шагам основного потока (1–18): расширение `Nа.` — альтернативный исход шага N. Все исключения, связанные с подписанием, формулируем как **события из ЭДО** — мы не наблюдаем причину напрямую, видим только статус. В плане фиксируем заголовки ситуаций; полные подшаги (`Nа1.`, `Nа2.` …) пишем в самом файле UC.

Имена статусов в расширениях — из расширенного enum концептуальной модели: `отклонен клиентом`, `отклонен парковкой`, `просрочен` для терминальных неуспешных веток.

**SLA подписи**: в `10б` и `12б` используем **7 календарных дней** на каждый этап подписи (клиент → Владелец). Заказчик SLA не озвучивал; число выбрано командой 2026-05-02 на основе типичной практики российских ЭДО (Диадок/СБИС: 5–10 дней). Параметр выносится в конфигурацию, при появлении заказчика — уточнить.

- **3а.** Идемпотентность по бизнес-ключу: если у клиента уже есть `[договор]` в статусе `черновик`, `на согласовании`, `на подписании клиентом` или `на подписании парковкой` с тем же `шаблономДоговораИд` и пересекающимся составом ПМ за пересекающийся период — Система не создает новый, а возвращает существующий с его текущим статусом.
- **6а.** ЭДО недоступно при отправке документа (сетевая ошибка / таймаут запроса) — `[договор]` остается в `черновик`, клиент уведомляется об ошибке отправки.
- **7а.** Подтверждение приема от ЭДО не получено (таймаут ответа / некорректный ответ без идентификатора документа) — `[договор]` остается в `черновик`, клиент уведомляется об ошибке отправки.
- **10а.** Из ЭДО получено событие отказа клиента от подписания — `[договор].статус` = `отклонен клиентом`, клиент уведомляется.
- **10б.** Событие подписания клиентом не получено в течение 7 календарных дней с момента отправки в ЭДО — `[договор].статус` = `просрочен`, клиент уведомляется.
- **12а.** Из ЭДО получено событие отказа Владельца от подписания — `[договор].статус` = `отклонен парковкой`, клиент уведомляется.
- **12б.** Финальный документ не получен в течение 7 календарных дней с момента подписи клиента — `[договор].статус` = `просрочен`, клиент уведомляется.

## Scope

**Входит:**

- Один UC-документ по структуре uc-10-2 (10 разделов, без дублирования архитектурного допущения).
- 6 решений из шапки плана отражены в потоке/расширениях.
- Основной поток (~18 шагов, принцип «один шаг — одно действие») + 7 расширений (`3а`, `6а`, `7а`, `10а`, `10б`, `12а`, `12б`).
- Расширение enum «Статус договора» и атрибутов сущности «Договор» в [концептуальной модели](../docs/artifacts/conceptual-model-with-attributes.md) — выполнено в этой же фазе.
- **Параллельная задача 1Б** (см. ниже): синхронизация ERD `contracts` с расширенным enum концептуальной модели.

**Не входит:**

- Детализация криптомеханики КЭП (HSM, КриптоПро) — упоминается только как факт.
- Конкретный вендор ЭДО (СБИС vs Диадок) — UC написан в нейтральных терминах «оператор ЭДО».
- ФТ, Sequence, JSON, XML, регламент — другие фазы.
- Правка реестра UC и диаграммы UC (UC-8.2 уже в реестре).
- TO-BE state machine договора с расширенным enum — артефакт **не трогаем** (договоренность с автором плана 2026-05-02).

**Запускается параллельно с потоком 1.1–1.7** (отдельным агентом в отдельном чате). Зависимостей с UC-документом нет: оба потока опираются на уже расширенную концептуальную модель (Шаг 1.0). Должна завершиться **до Фазы 4 (Sequence)** — там потребуется консистентность ERD с моделью.

### Файл

### Что сделать

1. **Расширить `contract_status_enum`** (строка ~1126, секция `### contracts`).

   Сейчас: `DRAFT, ACTIVE, EXPIRED, TERMINATED`.

   Стало (10 значений с маппингом на расширенную концептуальную модель):

   | ENUM (ERD)                  | Концептуальная модель     |
   | --------------------------- | ------------------------- |
   | `DRAFT`                     | `черновик`                |
   | `UNDER_REVIEW`              | `на согласовании`         |
